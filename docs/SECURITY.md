# Безопасность

Практические правила: хранение секретов, доступ к компонентам, что не коммитить. Детальное расследование прошлых инцидентов — [SECURITY_INVESTIGATION.md](SECURITY_INVESTIGATION.md).

---

## Секреты и конфигурация

- **Все секреты — только в `.env`.** Не коммитить `.env` в репозиторий. Справочник переменных — `env.example` (без реальных ключей).
- **Пароли:** использовать стойкие пароли для PostgreSQL (`POSTGRES_PASSWORD`), Redis (`REDIS_PASSWORD`), админки (`ADMIN_PASSWORD`). В проде не использовать дефолты вроде `trends`.
- **API-ключи:** Telegram (`TELEGRAM_BOT_TOKEN`), провайдеры генерации (`GEMINI_API_KEY`, `OPENAI_API_KEY` и т.д.), ЮKassa — только в переменных окружения, не в коде и не в git.
- **Админка:** авторизация по логину/паролю (JWT или session). Пароль админки не хранить в открытом виде в логах; при компрометации — сменить и перезапустить API.

---

## Сеть и порты

- **PostgreSQL и Redis** в проде должны слушать только localhost (127.0.0.1) или внутреннюю сеть, не 0.0.0.0 в интернет. Иначе возможны несанкционированный доступ и атаки (см. SECURITY_INVESTIGATION).
- **API (8000)** и **админка (3000)** — публичные для пользователей и бота; перед ними желательно reverse proxy (nginx/traefik) с HTTPS.
- **Cleanup (8001)** — только для админских операций; в проде привязать к 127.0.0.1 или защищать отдельно. Эндпоинт `/cleanup/run` должен требовать авторизацию (X-Admin-Key или аналог).

---

## Админка и API

- Все роуты админки под префиксом `/admin` защищены: JWT или session, проверка прав. Не отключать проверку авторизации для продакшена.
- CORS: задавать `CORS_ORIGINS` только доверенными доменами админки, не `*` в проде.
- Логи: не логировать тело запросов с токенами и паролями; при ошибках платежей — не выводить полные номера карт и PII в открытом виде.

---

## Данные пользователей

- Персональные данные (telegram_id, username, фото) хранятся в БД и на диске; доступ только у сервисов приложения и админов через админку.
- Команда/кнопка «Удалить мои данные» (`/deletemydata`) должна приводить к удалению или анонимизации данных пользователя в соответствии с политикой. Реализация — в боте и при необходимости в воркерах очистки.
- Бэкапы БД и файлов с пользовательскими данными хранить в защищённом месте с ограниченным доступом.

---

## Чек-лист перед деплоем

- [ ] `.env` не попал в репозиторий; все секреты заданы в окружении деплоя.
- [ ] Пароли БД и Redis не дефолтные; Redis с паролем.
- [ ] PostgreSQL и Redis не слушают на 0.0.0.0 в интернет (только localhost или внутренняя сеть).
- [ ] Админка доступна только по HTTPS (или через VPN) и с ограниченным CORS.
- [ ] URL webhook'а ЮKassa указывает на ваш API с HTTPS; при необходимости — проверка подписи/источника запроса.
