# Разработка и контрибьюция

Как разрабатывать: тесты, миграции, код-стайл.

---

## Окружение и запуск

- Локальный запуск: см. [GETTING_STARTED.md](GETTING_STARTED.md). Конфигурация — `.env` из `env.example`; секреты не коммитить.
- Бэкенд: Python 3.12, зависимости в `requirements.txt` (или аналог). API: `uvicorn` на порту 8000, бот и воркеры — через `docker compose` или локально по необходимости.

---

## Тесты

- Тесты лежат в каталоге **`tests/`**: `tests/api/`, `tests/services/`, `tests/paywall/`, `tests/photo_merge/`, `tests/referral/` и др.
- Запуск (из корня проекта):  
  `python -m pytest tests/`  
  или с опциями:  
  `python -m pytest tests/ -v --tb=short`  
  Для конкретного файла:  
  `python -m pytest tests/api/test_admin_audit.py -v`
- Убедитесь, что установлены зависимости для тестов (pytest и др., если вынесены в dev-зависимости).
- Перед коммитом желательно прогнать тесты в затронутых областях (API, сервисы платежей, paywall).

---

## Миграции БД

- Миграции — SQL-файлы в каталоге **`migrations/`** (и подкаталог `migrations/seed/`). Нумерация по порядку: `001_...`, `002_...`, и т.д.
- При старте через `./start.sh` скрипт по очереди применяет миграции к БД в контейнере (см. `start.sh`). На проде миграции обычно применяют вручную или отдельным шагом деплоя.
- Добавление новой миграции: создать файл `migrations/NNN_short_description.sql` с совместимым SQL (ALTER TABLE, CREATE INDEX и т.д.), проверить на копии БД. Не удалять и не переименовывать уже применённые файлы.

---

## Код-стайл и линтеры

- Python: следовать стилю проекта (отступы, кавычки, длина строк). При наличии в проекте — использовать black, ruff, flake8 или pylint по конфигу репозитория.
- Type hints приветствуются для аргументов и возвращаемых значений публичных функций.
- Frontend (админка): код в `admin-frontend/`; при наличии ESLint/Prettier — запускать перед коммитом.

---

## Ветки и коммиты

- Работа в отдельной ветке; merge в main после ревью (если принято в команде). Осмысленные сообщения коммитов.
- Не коммитить секреты, `.env`, временные файлы с ключами. Проверить `git status` перед коммитом.
