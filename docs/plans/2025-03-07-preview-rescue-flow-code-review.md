# Code Review: Preview Rescue Flow (pre-prod)

## 1. Critical bugs

### 1.1 [FIXED] Аналитика блокировала основной флоу
- **Где:** `rescue_reject_set`, `rescue_reason_more`
- **Проблема:** При падении `ProductAnalyticsService.track()` (БД/Redis) пользователь не видел экран C1/E или не получал reroll — показывалась только «Ошибка».
- **Исправление:** Вызовы аналитики обёрнуты в `try/except: pass`, основной флоу не зависит от аналитики.

### 1.2 [FIXED] Reroll при удалённом исходном файле
- **Где:** `rescue_reason_more` (бот), `generate_take` (воркер)
- **Проблема:** Если исходный файл по `input_local_paths` удалён или недоступен, создавался Take и воркер падал/фейлил генерацию без понятного сообщения.
- **Исправление:** В боте перед созданием reroll Take проверяется `os.path.isfile(first_path)`; в воркере для TREND добавлена ранняя проверка `input_image_path` и явный fail с сообщением «Исходное фото недоступно».

## 2. High-risk issues

### 2.1 Race при двойном клике «Хочу еще варианты»
- **Где:** `rescue_reason_more`
- **Риск:** Два быстрых запроса могут оба пройти `has_reroll_for_trend == False` до коммита первого — создадутся два reroll Take, пользователь получит два бесплатных reroll.
- **Рекомендация:** Принять как редкий edge case или добавить блокировку по (session_id, trend_id) в Redis с TTL 5–10 сек перед созданием reroll.

### 2.2 Сессия в статусе completed
- **Где:** `rescue_other_photo` → `_rescue_save_photo_and_start_take`
- **Риск:** Между «Загрузить другое фото» и отправкой фото сессия может быть завершена (другой сценарий). Take создаётся с `session_id` завершённой сессии; квота не тратится (`is_rescue_photo_replace`), но семантика «активная сессия» размыта.
- **Рекомендация:** Опционально проверять `session.status == "active"` и при `completed` показывать «Сессия завершена. Начните с «Создать фото»» и очищать state.

### 2.3 Зависание в rescue_waiting_photo
- **Где:** Состояние `rescue_waiting_photo`
- **Риск:** Пользователь нажал «Загрузить другое фото», но не отправил фото. Остаётся в этом состоянии; при следующем «Создать фото» или других действиях контекст может быть неочевиден.
- **Смягчение:** Есть `rescue_waiting_photo_wrong_input` и `/cancel` очищает state. В `unknown_message` для этого состояния можно добавить явную подсказку «Отправьте фото или /cancel».

## 3. Logic problems

### 3.1 Нет проверки владельца при rescue:photo_tip / rescue:other_trend
- **Где:** `rescue_photo_tip`, `rescue_other_trend`
- **Проблема:** По callback передаётся только `take_id`. Любой пользователь, знающий take_id другого, может открыть экраны G/I (только инфо и кнопки). Критичного доступа к данным нет, но возможен лёгкий спам/чужие сценарии.
- **Рекомендация:** Для консистентности можно проверять `take.user_id == user.id` и при несовпадении отвечать «Нет доступа».

### 3.2 deliver_hd_one при fav_id is None
- **Статус:** Уже исправлено ранее — при `fav_id is None` показывается кнопка «Избранное», а не «Открыть в HD».

## 4. Edge cases missed

### 4.1 Длина callback_data (Telegram limit 64 bytes)
- **Проверка:** `rescue:reject_set:` + UUID (36) ≈ 53 байта. Остальные префиксы короче. **OK.**

### 4.2 Очистка входных файлов
- **Проверка:** Cleanup удаляет только `Job.input_local_paths`, не Take. Входные файлы Take не трогаются. **OK.**

### 4.3 Поведение при падении send_task после commit
- **Сценарий:** В `rescue_reason_more` и `_rescue_save_photo_and_start_take` после `db.commit()` вызывается `celery_app.send_task()`. Если Redis/очередь недоступны, Take уже создан и остаётся в статусе `generating`.
- **Рекомендация:** Существующая общая проблема (не только rescue). Имеет смысл иметь джобу/мониторинг «зависших» Take в `generating` и либо ретрай, либо ручной разбор.

### 4.4 FSM state после исключения в rescue_other_photo
- **Где:** `rescue_other_photo` — `state.set_state` и `state.update_data` внутри `try`, затем `callback.message.answer`.
- **Проблема:** Если `answer` выбросит (сеть, лимиты Telegram), state уже установлен, пользователь не увидит «Отправьте новое фото» и может отправить фото позже — тогда флоу отработает. Если же падение до `set_state`, state не меняется. **Приемлемо.**

## 5. What must be fixed before release

**Обязательно (сделано в этом ревью):**
1. Аналитика не должна блокировать показ экранов C1/E и создание reroll — обёрнута в try/except.
2. Проверка существования файла перед созданием reroll Take в боте.
3. Ранний fail в `generate_take` для TREND при отсутствии `input_image_path` с понятным сообщением пользователю.

**Желательно перед продом:**
4. В `rescue_photo_tip` и `rescue_other_trend` проверять, что `take.user_id == current user.id`, и при несовпадении отвечать «Нет доступа».
5. В `_rescue_save_photo_and_start_take` при `session.status != "active"` не создавать Take, а показать «Сессия завершена» и очистить state.

**Можно отложить:**
6. Защита от двойного клика «Хочу еще варианты» (Redis lock или повторная проверка после создания Take).
7. В `unknown_message` для состояния `rescue_waiting_photo` — явная подсказка «Отправьте фото или /cancel».
