# Preview Rescue Flow — дизайн

## Цель

Помочь пользователю найти удачный результат внутри выбранного тренда, не превращая продукт в бесконечную бесплатную генерацию. Пользователь может сразу отказаться от всего сета из 3 превью; один бесплатный reroll сета в рамках тренда; после второго отказа — направление в замену фото / другой тренд.

## Текущее состояние

- **Первый показ превью**: воркер `generate_take` после успешной генерации отправляет `send_media_group` (3 фото) и сообщение «Выберите лучший вариант» с кнопками `⭐ A | ⭐ B | ⭐ C`, `📸 Ещё фото`, `📋 Избранное`.
- **Выбор A/B/C**: хендлер `choose_variant` (callback `choose:{take_id}:{variant}`) добавляет вариант в избранное, показывает «Вариант X в избранном» и либо paywall (free), либо кнопку «Забрать 4K».
- **Take**: есть поле `is_reroll` (Boolean). Сессия считает квоту по `takes_used`; `use_take(session)` вызывается в `generate_take` при успехе.
- Кнопки «Все 3 не подходят» и rescue-экраны отсутствуют.

## Ключевые решения

1. **Раунд** определяется по Take: `is_reroll=False` → round 1, `is_reroll=True` → round 2. Отдельное поле «round» в БД не вводим.
2. **Reroll не тратит квоту**: при создании rescue-reroll Take выставляем `is_reroll=True`; в `generate_take` при успехе вызываем `use_take(session)` только если `not take.is_reroll`.
3. **Один бесплатный reroll на тренд в рамках сессии**: перед созданием reroll Take проверяем, что в этой сессии для этого `trend_id` ещё нет Take с `is_reroll=True`. Если уже есть — не создаём второй reroll, показываем экран E.
4. **Замена фото**: лимит «1 замена фото на тренд» (по ТЗ «ограничена»). Реализация: в FSM храним `rescue_photo_replaced_trends: set` или при выборе «Загрузить другое фото» проверяем счётчик замен по сессии+тренду; при первой замене — разрешаем и помечаем. Упрощение: один раз за сессию для текущего тренда (хранить в state `rescue_photo_used_for_trend_id`).
5. **Тексты и кнопки** — по ТЗ (см. экраны ниже). Шаблоны можно вынести в `defaults`/БД позже; первый этап — литералы в коде.

## Callback_data и FSM

- `rescue:reject_set:{take_id}` — «Все 3 не подходят»; хендлер загружает Take, смотрит `is_reroll`: false → C1, true → E.
- `rescue:reason:face:{take_id}`, `rescue:reason:style:{take_id}`, `rescue:reason:more:{take_id}` — с C1.
- `rescue:other_photo:{take_id}`, `rescue:other_trend:{take_id}` — с E (и при необходимости с C1 для style).
- `rescue:photo_tip` — с F → G.
- Для «Загрузить другое фото»: переходим в состояние `rescue_waiting_photo`, в data — `rescue_trend_id`, `rescue_session_id`, `rescue_take_id` (контекст). После получения фото — создаём новый Take (тот же trend_id, новые input_local_paths), не reroll (это новый вход), не тратим квоту только если лимит «1 замена» считаем как бесплатную замену (по ТЗ ограничить — считаем 1 замену на тренд; квоту не тратим при первой замене или тратим — уточнение в плане: **не тратим** 1 замену фото на тренд).
- «Попробовать другой тренд»: выходим в выбор трендов (тот же session, те же фото в FSM или session.input_photo_path для коллекции); без нового состояния, просто навигация.

## Изменения по файлам (сводка)

- **app/workers/tasks/generate_take.py**: текст и клавиатура в зависимости от `take.is_reroll`; для round 1 — добавить кнопку «🔁 Все 3 не подходят»; не вызывать `use_take` для `is_reroll=True`.
- **app/bot/main.py**: новые callback-хендлеры (rescue:reject_set, rescue:reason:*, rescue:other_photo, rescue:other_trend, rescue:photo_tip); экран B/D — унифицировать текст «Отлично. Можно открыть…» и одну кнопку «💎 Открыть в HD»; состояние `rescue_waiting_photo` и хендлер фото для rescue; после выбора A/B/C не показывать «Оцените результат» сразу под основным сообщением — по ТЗ выбор = принятие, сразу «Открыть в HD».
- **app/services/sessions/service.py** (опционально): метод `count_reroll_takes(session_id, trend_id)` для проверки «уже был reroll».
- **app/services/takes/service.py**: при создании rescue reroll — `create_take(..., is_reroll=True)` (если create_take не принимает is_reroll — выставить на take после create).

## Ограничения

- Reroll: максимум 1 Take с `is_reroll=True` на пару (session_id, trend_id).
- Замена фото: 1 раз на тренд в рамках сессии (храним в FSM или в Session/отдельная таблица; минимально — в FSM state `rescue_photo_used_trends: [trend_id]`).

## Аналитика

- Логировать/трекать: `rescue_reject_set` (take_id, round), `rescue_reason_face/style/more`, `rescue_reroll_started`, `rescue_photo_uploaded`, `rescue_other_trend` — для метрик конверсии и причин отказа.
