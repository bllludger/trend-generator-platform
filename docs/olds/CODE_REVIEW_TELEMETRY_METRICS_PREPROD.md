# Code review: метрики производительности + среднее время 3 снимков (перед продом)

## Обзор фичи

- **Бэкенд:** в `GET /admin/telemetry` добавлены глобальные `takes_succeeded`, `takes_failed` и расчёт `take_avg_generation_sec` по событиям AuditLog `take_previews_ready` + Join с Take.
- **Фронт (TelemetryPage):** карточки «За 24ч», «Успешно», «Ошибки» переведены на учёт Job + Take; добавлена карточка «Сред. время 3 снимков».

---

## 1. Critical bugs

**Не найдено.** Нет падений, неверных типов в ответе API и явно сломанных сценариев.

---

## 2. High-risk issues

### 2.1 ~~Среднее время: дубликаты~~ (исправлено)

При ретрае задачи Celery один и тот же Take мог получать два события `take_previews_ready`. В коде теперь берётся **одно значение на Take** (первое событие по `entity_id`), затем усредняются только эти длительности. Дубликаты больше не искажают метрику.

### 2.2 ~~Отрицательная длительность~~ (исправлено)

Отрицательные длительности (при сдвиге часов или баге) больше не попадают в расчёт: в среднее идут только `sec >= 0`.

---

## 3. Logic problems

### 3.1 «В очереди» — только Job

Карточка «В очереди» берёт `queue_length` из API — это только Job в статусах CREATED/RUNNING. Take в статусе `generating` в эту очередь не входят. Для консистентности с остальными карточками (Job + Take) можно либо оставить как есть и считать это «очередь Celery (Job)», либо добавить в API счётчик Take со статусом `generating` и показывать сумму. Текущее поведение не баг, но подпись «Celery queue» может вводить в заблуждение, если ожидают «все в работе».

### 3.2 Success rate и error rate не дополняют друг друга до 100%

В окне есть задачи «в работе» (Job CREATED/RUNNING, Take generating). Поэтому `succeeded + failed < total` — норма. Success rate = succeeded/total, error rate = failed/total; они не обязаны давать 100% в сумме. Логика корректна, регрессии нет.

---

## 4. Edge cases missed

### 4.1 Очень большое окно и память при расчёте среднего

При `window_hours=720` запрос `take_avg_q` тянет все пары (audit.created_at, take.created_at) за 30 дней. При высокой нагрузке это десятки тысяч строк в памяти. Сейчас `window_hours` ограничен `le=720`, поэтому риск ограничен, но при росте нагрузки стоит считать среднее в БД (например, через подзапрос с группировкой по take id и AVG по длительностям), чтобы не материализовать все строки в Python.

### 4.2 Take без trend_id уже учтены в глобальных счётчиках

Глобальные `takes_succeeded` и `takes_failed` считаются по всем Take за окно (включая без trend_id). Согласованность с `takes_window` (тоже все Take) соблюдена, edge case обработан.

### 4.3 Отображение «Сред. время 3 снимков» при 0 событий

При отсутствии событий `take_previews_ready` за окно API возвращает `take_avg_generation_sec: null`. На фронте проверка `data?.take_avg_generation_sec != null` даёт отображение «—». Корректно.

### 4.4 Целочисленность и форматирование времени

Для времени < 60 сек используется `Math.round(sec)` (целые секунды), для ≥ 60 — `(sec/60).toFixed(1)` мин. Отрицательные или NaN на фронт не передаём при условии исправления п. 2.2 на бэкенде.

## 6. Final verdict

**Ready** для прода.

Критичных багов нет. Учёт Job + Take в карточках и расчёт успехов/ошибок согласованы. Среднее время 3 снимков считается по одному значению на Take, отрицательные длительности отфильтрованы. Оставшееся замечание (подпись «В очереди») — по желанию.
