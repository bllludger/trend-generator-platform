# Code review: фича «Тренды — аналитика» (перед продом)

**Дата:** 2026-03-10  
**Область:** бэкенд `GET /admin/trends/analytics`, фронт TrendsAnalyticsPage, дашборд (удаление блока «Топ трендов за 30 д»).

---

## 1. Critical bugs

### 1.1 Нет обработки ошибки запроса на странице аналитики

**Файл:** `admin-frontend/src/pages/TrendsAnalyticsPage.tsx`

При падении API (500, сеть, таймаут) `useQuery` переводит запрос в состояние `isError: true`, `data: undefined`. Компонент при этом не проверяет `isError` и рендерит страницу с пустыми данными: пустая таблица, пустая диаграмма, блок «Топ трендов» с текстом «Пока нет активности по трендам». Подписи карточек при этом показывают «за всё время» (`periodLabel` из-за `data?.window_days == null` даёт «всё время»).

**Итог:** Пользователь не понимает, что произошла ошибка, и думает, что за период просто нет данных.

**Исправление:** Добавить обработку `isError` (и при необходимости `isLoading` при refetch): показывать блок с сообщением об ошибке и кнопкой «Повторить» (например, через `refetch()`), и не показывать контент с «Нет данных» как при успешном пустом ответе.

## 2. High-risk issues

### 2.1 Возможное отображение "NaN" в тултипе диаграммы

**Файл:** `admin-frontend/src/pages/TrendsAnalyticsPage.tsx`, кастомный `Tooltip`.

Используется `formatNumber(Number(payload[0].value))`. Если по какой-то причине `payload[0].value` будет `undefined` (например, смена структуры данных Recharts или баг), `Number(undefined)` даёт `NaN`, а `Intl.NumberFormat().format(NaN)` в части окружений выводит строку `"NaN"`.

**Исправление:** Приводить к числу с запасным значением, например:  
`formatNumber(Number(payload[0]?.value ?? 0))`.

### 2.2 Пустой список трендов на бэкенде: лишняя нагрузка и неоднозначность

**Файл:** `app/api/routes/admin.py`, `admin_trends_analytics`.

При `svc.list_all()` → `all_trends = []` получаем `trend_ids = []`. Запросы с условием `Job.trend_id.in_([])` и `Take.trend_id.in_([])` в разных версиях SQLAlchemy/БД могут вести себя по-разному (в т.ч. не «ни одной строки»). Даже при корректном «0 rows» выполняются три тяжёлых запроса (Job, Take, AuditLog) без реальной пользы.

**Исправление:** В начале эндпоинта добавить ранний выход:

```python
if not trend_ids:
    return {"window_days": window_days if window_days else None, "items": []}
```

Так избегаем неоднозначности и лишней нагрузки при нуле трендов.

## 3. Logic problems

### 3.1 Подпись периода при ошибке

**Файл:** `admin-frontend/src/pages/TrendsAnalyticsPage.tsx`.

`periodLabel` вычисляется как  
`data?.window_days == null || data?.window_days === 0 ? 'всё время' : \`${data.window_days} д.\``  
При ошибке `data` — `undefined`, поэтому `periodLabel` всегда «всё время». Карточки тогда подписаны как «Топ трендов за всё время» и «Все тренды за всё время» даже когда показывается пустое/ошибочное состояние.

**Исправление:** Использовать `periodLabel` только при успешном ответе (например, когда `!isError && data != null`), а в состоянии ошибки показывать нейтральную подпись или явное «Ошибка загрузки» без «за всё время».

### 3.2 Логика «Топ трендов» списком vs таблица

Блок «Топ трендов» списком берёт до 10 трендов с активностью `(jobs_total + takes_total) > 0 || (chosen_total ?? 0) > 0` из уже отсортированного по убыванию объёма `items`. Таблица ниже показывает все тренды из того же `items`. Порядок и критерии согласованы, дублирования по смыслу нет — логика корректна.

## 4. Edge cases missed

- **Пустой ответ API:** `items: []` — обрабатывается: пустая таблица, «Нет данных за период» в диаграмме, «Пока нет активности» в блоке топа. Достаточно.
- **Очень длинное имя тренда:** обрезка до 25 символов + «…» в данных для BarChart; в тултипе показывается `fullName`. Приемлемо.
- **Тренд без эмодзи:** используется `row.emoji || '—'` в таблице и в списке; в `chartData` — `i.emoji ? i.emoji + ' ' : ''`. Ошибок нет.
- **Backend: `window_days=0`:** `since = None`, фильтры по дате не накладываются, в ответе `window_days: null` — поведение соответствует задумке «всё время».
- **Дашборд:** второй запрос `getDashboard(DASHBOARD_HISTORY_DAYS * 24)` остаётся нужным для блока «Статусы задач (30 д)» и снимков за 30 д. Регрессий по данным нет. Ссылка «Аналитика по трендам» ведёт на `/trends-analytics`, маршрут в приложении есть.

## 5. What must be fixed before release

Обязательно перед выкладкой в прод:

1. **Обработка ошибки запроса на TrendsAnalyticsPage**  
   Использовать `isError` (и при необходимости `refetch`): при ошибке не рендерить «пустые данные», а показывать явное сообщение об ошибке и возможность повторить запрос.

2. **Защита от NaN в тултипе диаграммы**  
   В кастомном Tooltip использовать `Number(payload[0]?.value ?? 0)` (или аналог), чтобы не показывать «NaN».

3. **Ранний выход при пустом списке трендов в API**  
   В `admin_trends_analytics` при `not trend_ids` сразу возвращать `{"window_days": ..., "items": []}` и не выполнять запросы к Job/Take/AuditLog.

Желательно:

4. Не использовать подпись «за всё время» в карточках, когда отображается состояние ошибки (показывать ошибку и не подставлять `periodLabel` как при успешном ответе).

## 6. Final verdict

**Not ready** для продакшена без исправлений.

После внесения пунктов 1–3 из раздела 5 фича может быть допущена к релизу; пункт 4 улучшит понятность интерфейса при сбоях.

## Исправления (применены 2026-03-10)

1. **Обработка ошибки запроса:** на TrendsAnalyticsPage добавлены `isError`, `refetch`; при ошибке показывается карточка с сообщением и кнопкой «Повторить».
2. **Тултип диаграммы:** используется `Number(payload[0]?.value ?? 0)` для защиты от NaN.
3. **Ранний выход при пустом списке трендов:** в `admin_trends_analytics` при `not trend_ids` сразу возвращается `{"window_days": ..., "items": []}`.
4. **periodLabel при ошибке:** подпись периода вычисляется только при наличии `data`; в заголовках карточек добавлен fallback `periodLabel || 'выбранный период'`.
