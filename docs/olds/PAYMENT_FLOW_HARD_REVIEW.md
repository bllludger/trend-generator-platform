# Жёсткий ревью Payment Flow — критические и высокорисковые проблемы

**Дата:** 2025-03-07  
**Область:** end-to-end сценарий оплаты (Stars, ЮMoney, unlock, session/upgrade), идемпотентность, выдача после оплаты, edge cases.

---

## 1. Critical issues

### 1.1 Unlock: откат транзакции при падении send_document → потеря платежа и риск двойной оплаты

**Где:** `app/bot/main.py`, `handle_successful_payment`, блок `pack_id == "unlock"` (≈3774–3820).

**Суть:**  
`credit_tokens()` и `send_document()` выполняются в одной сессии БД (`with get_db_session() as db`). При успехе в конце блока вызывается `db.commit()`. Если `send_document()` выбрасывает исключение (сеть, Telegram API, таймаут), срабатывает `get_db_session`: `db.rollback()` — **запись Payment откатывается**. Пользователь уже списан Telegram’ом, но в БД платежа нет.

**Последствия:**
- Деньги списаны, выдачи нет, записи о платеже нет.
- При повторном нажатии «Разблокировать» пользователь получает новый инвойс (новый nonce → новый payload), платит снова. Второй платёж может пройти целиком → **двойная оплата за один unlock**.

**Нужно:**  
Фиксировать платёж до отправки файла (отдельный commit после `credit_tokens` или отдельная сессия с commit), затем пытаться доставку; при падении доставки — не откатывать платёж, логировать и доставлять позже (retry/очередь/поддержка). Либо явно пытаться refund при исключении только если платёж точно не был закоммичен (сложно без явного двухфазного шага).

---

### 1.2 Unlock: нет защиты от повторной оплаты за один и тот же job

**Где:**  
- Отправка инвойса: `unlock_photo()` (≈3389–3426) — не проверяет, есть ли уже успешный платёж за этот `job_id`.  
- Pre-checkout: `validate_pre_checkout()` для legacy unlock (≈334–360) — не проверяет «уже оплачено за этот job» / «job уже разблокирован».  
- Successful payment: перед `credit_tokens()` для unlock не проверяется `job.unlocked_at` и нет проверки «уже есть Payment с pack_id=unlock и job_id=…».

**Последствия:**
- После первой оплаты при падении доставки пользователь может снова нажать «Разблокировать», получить инвойс и оплатить второй раз → два платежа за один unlock.
- При повторной доставке сообщения `successful_payment` (редко, но возможно) возможна повторная запись платежа (другой charge_id) и повторная отправка файла.

**Нужно:**
- В **pre_checkout** для unlock: отклонять оплату, если для этого `job_id` уже есть успешный платёж (Payment с pack_id=unlock и job_id) или job уже разблокирован (`unlocked_at` / не is_preview).
- При **отправке инвойса** unlock: не показывать инвойс, если уже есть оплата за этот job; при оплате есть, а доставки не было — пытаться доставить (retry), не создавая новый платёж.
- В **successful_payment** для unlock: до вызова `credit_tokens` проверять «уже разблокирован / уже есть платёж за этот job»; при уже оплаченном — рефандить текущий charge_id (Stars) и сообщить «уже разблокировано»; не создавать второй Payment и не слать файл второй раз.

---

### 1.3 ЮMoney: trial_already_used без автоматического возврата

**Где:** `app/bot/main.py`, `handle_successful_payment`, ветка `yoomoney_session:` (≈3582–3611).

**Суть:**  
При гонке «Trial уже использован» после успешной оплаты ЮMoney вызывается только логирование и сообщение пользователю «обратитесь в поддержку — мы вернём средства». **Автоматического рефанда через провайдера нет** (в отличие от Stars, где вызывается `refund_star_payment`).

**Последствия:**  
Средства списаны с карты, продукт не выдан, возврат только вручную через поддержку/ЮKassa → риск потери денег и недоверия.

**Нужно:**  
Реализовать возврат через API ЮKassa (или используемый платёжный провайдер) при `trial_already_used` и логировать результат; при технической невозможности рефанда — явный алерт и процедура ручного возврата с трекингом.

---

### 1.4 Общий exception в successful_payment: оплата списана, выдачи/рефанда нет

**Где:** `app/bot/main.py`, `handle_successful_payment`, блок `except Exception` (≈3966–3971).

**Суть:**  
При любом исключении в обработчике (таймаут БД, ошибка в `credit_tokens`/session purchase, падение доставки и т.п.) пользователю показывается общее сообщение об ошибке и просьба написать в поддержку. **Рефанд в catch-all не делается.**

**Последствия:**  
Telegram уже списал Stars (или прошла оплата ЮMoney). При падении после списания пользователь остаётся без продукта и без автоматического возврата → «успешная оплата без выдачи» и потеря денег до ручного разбора.

**Нужно:**  
В блоке except для **Stars** по возможности вызывать `refund_star_payment(charge_id)` (если уверены, что начисление/выдача не были закоммичены), с логированием успеха/неудачи. Для **ЮMoney** — те же принципы, что и в п. 1.3. Разделить сценарии «точно не обработали» vs «неизвестно» (идемпотентность), чтобы не рефандить уже выданный заказ.

## 2. High-risk issues

### 2.1 Rate limit: fail open при недоступности Redis

**Где:** `app/services/payments/service.py`, `_check_rate_limit()` (≈596–605).

**Суть:**  
При `redis.RedisError` возвращается `True` («лимит не превышен») — покупка разрешается.

**Последствия:**  
При падении/перезапуске Redis лимит на число покупок (в т.ч. trial) не действует → возможен abuse (много trial, массовые покупки).

**Рекомендация:**  
Fail closed: при ошибке Redis возвращать `False` и отклонять pre_checkout с сообщением «Сервис временно недоступен» или ввести запасной лимит (например, in-memory/DB) для критичных сценариев.

---

### 2.2 Unlock: нет проверки «уже разблокирован» перед созданием Payment и доставкой

**Где:** Тот же блок unlock в `handle_successful_payment`.

**Суть:**  
Перед `credit_tokens()` не проверяется `job.is_preview` / `job.unlocked_at`. При повторной доставке `successful_payment` (дубль от Telegram или повтор запроса) возможна вторая запись Payment (другой charge_id при повторном инвойсе) и повторная отправка файла.

**Рекомендация:**  
В начале обработки unlock по `job_id` проверять: если job уже разблокирован — не вызывать `credit_tokens`, вызвать refund для текущего `charge_id` (Stars), ответить «уже разблокировано». Аналогично не создавать второй Payment при уже существующем Payment за этот job (unlock).

---

### 2.3 Сумма ЮMoney в pre_checkout и в инвойсе: жёстко заданный DISPLAY_RUB

**Где:**  
`app/services/payments/service.py`, `validate_pre_checkout` для `yoomoney_session:` (≈266–272);  
`app/services/balance_tariffs.py`, `DISPLAY_RUB`; формирование цен в боте при send_invoice (ЮMoney).

**Суть:**  
Ожидаемая сумма в pre_checkout считается из `DISPLAY_RUB.get(pack_id)` или из `pack.stars_price * star_to_rub`. Инвойс формируется так же. Если поменять только одно место (например, DISPLAY_RUB в коде) и забыть синхронизировать — возможен отказ pre_checkout или расхождение суммы.

**Рекомендация:**  
Один источник правды для суммы (например, Pack или конфиг с приоритетом), использование в и pre_checkout, и при создании инвойса; тесты на консистентность сумм.

## 3. Broken payment scenarios

| Сценарий | Что ломается | Итог |
|----------|----------------------|-----|
| Unlock: оплата прошла, send_document падает (сеть/API) | Rollback всей транзакции → Payment не сохраняется | Оплата есть, выдачи нет, записи нет; при повторе — двойная оплата |
| Unlock: пользователь дважды нажимает «Разблокировать» и дважды платит | Нет проверки «уже оплачено за job» в pre_checkout и при инвойсе | Два платежа за один unlock |
| ЮMoney: trial уже использован, оплата прошла | Нет автоматического рефанда | Деньги списаны, продукт не выдан |
| Любая ошибка в successful_payment после списания (БД, таймаут и т.д.) | Нет рефанда в общем except | Оплата без выдачи |
| Redis недоступен при покупке | Rate limit отключён (fail open) | Злоупотребление trial/покупками |

## 4. Missing safeguards

- **Unlock:** проверка «уже есть платёж за этот job» / «job уже разблокирован» в pre_checkout, при показе инвойса и в successful_payment.
- **Unlock:** разделение «сохранить платёж» и «доставить файл»: commit платежа до отправки файла; при падении доставки — не откатывать платёж, доставлять позже (retry/очередь/ручная выдача).
- **successful_payment:** в общем обработчике исключений — попытка рефанда (Stars/ЮMoney) при уверенности, что начисление/выдача не были закоммичены, с логированием.
- **ЮMoney trial_already_used:** автоматический возврат через API провайдера или явный процесс ручного возврата с трекингом.
- **Rate limit:** при недоступности Redis — fail closed или запасной лимит.
- **Идемпотентность unlock по job:** не создавать второй Payment для того же job_id (unlock); при повторном successful_payment для уже разблокированного job — рефанд + сообщение.
- **Следность:** логировать charge_id, job_id, user_id, момент commit платежа и результат доставки (успех/ошибка), чтобы можно было сверять с Telegram/ЮKassa и восстанавливать «оплатил, но не доставлено».

## 5. What must be fixed before prod

**Обязательно:**

1. **Unlock и транзакция**  
   Не откатывать запись Payment при падении send_document. Либо commit после `credit_tokens` в отдельной сессии/шаге, затем доставка; либо отдельная задача на доставку с retry. При ошибке доставки — не создавать новый платёж при повторном нажатии «Разблокировать».

2. **Unlock и повторная оплата**  
   - В pre_checkout для unlock: отклонять, если по job_id уже есть успешный платёж (pack_id=unlock) или job разблокирован.  
   - При показе инвойса unlock: не показывать инвойс, если уже оплачено за этот job; при «оплачено, но не доставлено» — только retry доставки.  
   - В successful_payment для unlock: до credit_tokens проверять «уже разблокирован / уже есть платёж за этот job»; при да — рефанд (Stars) и сообщение «уже разблокировано».

3. **ЮMoney trial_already_used**  
   Ввести автоматический возврат через API провайдера или документированный ручной процесс с алертами и трекингом; пользователю — однозначное сообщение.

4. **Общий except в successful_payment**  
   Для Stars: при исключении пытаться рефандить по charge_id, если можно гарантировать, что начисление/выдача не были закоммичены; логировать результат. Для ЮMoney — по возможности то же или явный ручной процесс.

**Желательно до prod:**

5. Rate limit при Redis down — не разрешать покупку (fail closed) или запасной лимит.  
6. Один источник правды для сумм ЮMoney (pre_checkout + инвойс).  
7. Логи и при необходимости аналитика: charge_id, job_id, user_id, commit платежа, результат доставки — для расследований и сверки с провайдером.

---

## 6. Final verdict

**Payment flow: not safe for production** в текущем виде.

Критичные риски:

- Потеря денег пользователя (оплата без выдачи, без рефанда при ошибках и при ЮMoney trial).
- Двойное списание (unlock при падении доставки и повторной оплате).
- Рассинхрон и silent failure (откат платежа при ошибке доставки, нет рефанда в except).

Перед выкладкой в prod необходимо закрыть пункты раздела 5 (обязательные исправления), после чего можно пересмотреть вердикт и при необходимости провести повторное ревью.
