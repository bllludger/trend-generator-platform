# Code Review: Кнопка «Забрать 4К» после активации пакета

**Фича:** после успешной оплаты пакета (Stars / YooMoney) показывать кнопку «💎 Открыть фото в 4K» и строку «Можно забрать последнее фото в 4K», если у пользователя есть избранный кадр с невыданной 4K.

**Файлы:** `app/services/favorites/service.py`, `app/bot/main.py`.

---

## 1. Critical bugs

### 1.1 Исключение в хелпере откатывает всю транзакцию (платёж теряется)

**Где:** `handle_successful_payment` → вызов `_pack_activated_message_and_keyboard(db, ...)` внутри `with get_db_session() as db:`.

**Суть:** Хелпер вызывается **до** выхода из `with get_db_session()`. Если внутри хелпера или при последующем `message.answer()` произойдёт исключение (например, сбой БД при `get_last_pending_hd_favorite`, или ошибка при отправке в Telegram), контекстный менеджер выполнит `db.rollback()` и повторно выбросит исключение. В результате откатятся **все** изменения в этой сессии БД, включая только что созданные `Payment`, `Session` и начисленный HD-баланс. Деньги у пользователя уже списаны (Stars/ЮMoney), а в нашей БД записи о платеже и сессии нет. Пользователь видит «⚠️ Произошла ошибка при обработке платежа» и остаётся без активированного пакета.

**Исправление (сделано):** В обоих местах (Stars и YooMoney) вызов `_pack_activated_message_and_keyboard` и `message.answer` обёрнуты в `try/except`. При исключении логируется `pack_activated_message_keyboard_error` и отправляется fallback: базовый текст + кнопка «В меню». Транзакция не откатывается.

---

## 2. High-risk issues

### 2.1 Нет явной проверки владельца при показе кнопки

**Где:** `_pack_activated_message_and_keyboard` строит `callback_data=f"deliver_hd_one:{last_pending.id}"` по `user.id` из той же сессии БД.

**Суть:** Кнопка привязана к `last_pending`, который получен по `user_id` (User.id по telegram_id). То есть мы показываем кнопку только для своего пользователя. Уязвимости «подставить чужой fav_id» в этом экране нет: callback_data формируется на сервере из данных БД. Проверка владельца уже есть в `deliver_hd_one_callback`: `if not fav or str(fav.user_id) != str(user.id)`. Риск не критичный, но стоит держать в голове: любая будущая логика, которая подставляет `fav_id` извне, должна сохранять эту проверку.

**Рекомендация:** оставить как есть; при любых доработках callback не ослаблять проверку в `deliver_hd_one_callback`.

### 2.2 Длина callback_data

**Где:** `deliver_hd_one:{fav_id}` (UUID — 36 символов, префикс 15).

**Суть:** Итого 51 байт. Лимит Telegram для `callback_data` — 64 байта. Запас есть, риск низкий.

## 3. Logic problems

### 3.1 «Последний» избранный может быть в статусе rendering

**Где:** `get_last_pending_hd_favorite` фильтрует по `hd_status != "delivered"`, то есть возвращает и записи со статусом `"rendering"`.

**Суть:** Если пользователь уже нажал «Открыть фото в 4K» по этому же избранному (например, из предыдущего сообщения), воркер выставил `hd_status = "rendering"`. После оплаты мы снова показываем кнопку для того же избранного (оно по-прежнему «последнее» с невыданной 4K). При нажатии `deliver_hd_one_callback` ответит «4K уже выдан или в обработке». Логика не ломается, но UX дублирует кнопку для уже обрабатываемого кадра.

**Исправление (сделано):** В `get_last_pending_hd_favorite` фильтр заменён на `Favorite.hd_status == "none"`. Кнопка показывается только для кадров, готовых к выдаче.

## 4. Edge cases missed

### 4.1 pack.emoji / pack.name

**Где:** вызов `_pack_activated_message_and_keyboard(db, telegram_id, pack.emoji, pack.name, remaining_display)`.

**Суть:** В модели `Pack`: `name` — `nullable=False`, `emoji` — `default=""`. Теоретически старые записи или админка могли оставить пустой `emoji`; тогда текст будет «Пакет  Trial» (двойной пробел). Не баг, но при желании можно подставлять `pack.emoji or ""` и обрезать лишние пробелы или задать fallback-эмодзи в одном месте.

### 4.2 user is None в хелпере

**Где:** `_pack_activated_message_and_keyboard` → `user = db.query(User).filter(User.telegram_id == telegram_id).one_or_none()`.

**Суть:** После только что выполненного `process_session_purchase*` пользователь обычно уже есть. Если по какой-то причине `user` не найден, хелпер не добавляет строку про 4K и не добавляет кнопку — остаётся только «В меню». Поведение безопасное и корректное.

### 4.3 Двойное нажатие «Открыть фото в 4K»

**Где:** пользователь быстро дважды нажимает кнопку после активации пакета.

**Суть:** Первый клик запускает `deliver_hd`, воркер ставит `hd_status = "rendering"`. Второй клик получает «4K уже выдан или в обработке». Регрессии нет, идемпотентность и защита от двойной выдачи сохранены.
