# Code review: централизация платежей и исторический график (перед продом)

## 1. Critical bugs

### 1.1 (ИСПРАВЛЕНО) Отсутствовал `db.commit()` в `payments_refund`
- **Где:** `app/api/routes/admin.py`, эндпоинт `POST /admin/payments/{payment_id}/refund`.
- **Суть:** После успешного `process_refund()` сессия не коммитилась. При закрытии сессии в `get_db()` изменения откатывались — возвраты не сохранялись в БД.
- **Исправление:** После `process_refund` добавлен вызов `db.commit()`.

---

## 2. High-risk issues

### 2.1 Unlock: пользователь удалён до вебхука
- **Где:** Вебхук ЮKassa → `mark_paid` → `record_yookassa_unlock_payment(order)`. Если пользователь по `order.telegram_user_id` не найден, метод возвращает `None`, но вебхук всё равно делает `commit` и ставит доставку в очередь.
- **Следствие:** Заказ помечается оплаченным, файл уходит в доставку, но запись в `payments` не создаётся → занижение выручки и рассинхрон отчётов.
- **Рекомендация:** Оставить текущее поведение (доставка важнее отчёта), но поднять уровень лога до `error` и завести алерт на `record_yookassa_unlock_user_not_found`.

### 2.2 GET /admin/payments/history без ограничения дат («Всё время»)
- **Где:** Фронт при выборе «Всё время» не передаёт `date_from`/`date_to`; бэкенд отдаёт все дни за всю историю.
- **Следствие:** При большом объёме платежей — тяжёлый запрос, риск таймаута и высокая нагрузка на БД.
- **Рекомендация:** Либо ограничить период по умолчанию (например, макс. 365 дней), либо ввести лимит по числу точек в ответе и документировать ограничение.

### 2.3 Гонка при двойном вебхуке unlock
- **Где:** Два одновременных запроса за один и тот же `payment_id`: оба проходят проверку `order.status in ("created", "payment_pending")`, оба вызывают `mark_paid` и `record_yookassa_unlock_payment`. Второй может получить `IntegrityError` (дубликат `telegram_payment_charge_id`).
- **Исправление:** В `record_yookassa_unlock_payment` добавлена обработка `IntegrityError`: `rollback` и повторный `raise`, чтобы вебхук вернул 5xx и ЮKassa повторил запрос. При повторе заказ уже будет в статусе `paid`, повторная вставка не выполнится.
- **Итог:** Поведение корректное, дубликатов платежей не будет.

## 3. Logic problems

### 3.1 Рефанд для платных выдач админом (admin_manual)
- **Где:** Платежи с `telegram_payment_charge_id.startswith("admin_manual:")` не отфильтрованы в `payments_refund`. В списке они отображаются как Stars, кнопка «Возврат» доступна только для `payment_method === 'stars'`, так что в UI возврат по ним возможен.
- **Следствие:** Возврат по «ручной» выдаче пакета списывает токены/сессию у пользователя, хотя реальных денег не было.
- **Рекомендация:** В эндпоинте рефанда добавить проверку на `admin_manual:` и возвращать 400 с пояснением, что возврат для ручной выдачи недоступен.

### 3.2 Формат даты при `granularity=week` в history
- **Где:** `date_trunc('week', ...)` в PostgreSQL возвращает `timestamp`; в коде делается `dt.date()` перед `isoformat()`, поэтому в ответе всегда `YYYY-MM-DD`. Логика корректна, проблем не выявлено.

### 3.3 Конверсия в stats (conversion_rate_pct)
- **Где:** В ответе `payments/stats` поле `conversion_rate_pct` всегда 0 (захардкожено).
- **Суть:** Не баг этой фичи, но метрика вводит в заблуждение. Имеет смысл либо считать конверсию (например, покупатели/уникальные пользователи за период), либо не отдавать поле.

## 4. Edge cases missed

### 4.1 Пустой `order.yookassa_payment_id`
- **Где:** `record_yookassa_unlock_payment`: при `not order.yookassa_payment_id` возвращается `None`.
- **Факт:** В сценарии вебхука `payment.succeeded` заказ уже в статусе `payment_pending` и `yookassa_payment_id` у него задан. Крайний случай — старые записи без ID; тогда вебхук не создаёт Payment, но и не падает. Оставить как есть, при необходимости дообогатить логирование.

### 4.2 История при отсутствии платежей за период
- **Где:** `payments/history` при пустом наборе данных возвращает `{"series": []}`. Фронт показывает «Нет данных за выбранный период.» — поведение корректное.

### 4.3 Фильтр по `pack_id` в history
- **Где:** При переданном `pack_id` оба запроса (основной и по `by_pack`) используют один и тот же отфильтрованный `q`, поэтому разбивка по пакетам за день соответствует отфильтрованным данным. Логика верна.

## 5. What must be fixed before release

1. **Сделано:** Добавить `db.commit()` в `payments_refund` после успешного `process_refund`.
2. **Сделано:** Обработать `IntegrityError` в `record_yookassa_unlock_payment` (rollback + raise), чтобы при гонке вебхук не оставлял сессию в неконсистентном состоянии и давал повторить запрос.
3. **Рекомендуется перед/вскоре после релиза:**
   - Запретить рефанд для `admin_manual` в `payments_refund` (проверка по `telegram_payment_charge_id`).
   - Ограничить запрос history при отсутствии дат (макс. период или макс. число точек) или явно задокументировать риск при «Всё время».
   - Повысить уровень лога при `record_yookassa_unlock_user_not_found` до error и при необходимости настроить алерт.

---

## 6. Стабильность и безопасность

- **Ошибки вебхука:** Исключения перехватываются, делается `rollback`, возвращается 200 — повторные попытки ЮKassa не зацикливаются.
- **Ошибки history:** Нет явного try/except; при сбое БД клиент получит 5xx. Для read-only эндпоинта допустимо.
- **Авторизация:** `payments_refund` и `payments/history` за защищённым роутом admin (`get_current_user`), доступ только для авторизованных админов.
- **Регрессии:** Расширение `_payment_method_from_row` и фильтров по способу оплаты обратно совместимо; старые записи без новых префиксов продолжают классифицироваться как stars. Расчёт выручки в stats (включение всех `amount_kopecks`) корректен.

---

## 7. Final verdict

**Not ready** до внесения критического исправления (commit в refund).  
**Ready** после внесённых правок (commit + обработка IntegrityError), при условии принятия рисков по п. 2.1 и 2.2 и выполнения рекомендаций из п. 5 (желательно до или сразу после релиза).

Итог: **критический баг с рефандом исправлен, гонка в вебхуке обработана. Выпуск в прод допустим с оговорками выше; оставшиеся пункты — улучшения и снижение рисков.**
