# Аудит интерфейса: Аналитика по трендам

По принципам **interface-design** (craft, consistency, token architecture). Объект: страницы «Все тренды за 30 д.», «Топ трендов за 30 д.», «Аналитика по трендам» (TrendsAnalyticsPage) и связанные компоненты.

---

## 1. Резюме

- **Что ок:** Иерархия заголовок/легенда/таблица читаема. Есть hover у строк, скелетоны, пустые и ошибочные состояния. Эмодзи у трендов задают контекст. Семантика успех/ошибка понятна.
- **Главные провалы:** Нет зафиксированного intent и product signature; смешаны глубина (border + shadow); отступы не строго по сетке; успех/ошибка/предупреждение не идут из токенов; токены универсальные, не из «мира продукта».

---

## 2. Проверки Mandate

### Swap test
- Шрифт: замена на другой sans почти не изменит ощущение — типографика не стала выразительным выбором.
- Макет: заголовок + селектор, карточка с заголовком и описанием, таблица, список — типовой дашборд. Замена на другой шаблон даст похожее «админка» ощущение.
- **Вывод:** много решений по умолчанию, мало осознанного выбора.

### Squint test
- Заголовки жирные, подписи приглушённые, заголовки таблицы отделяются от данных — иерархия при «размытии» в целом видна.
- **Вывод:** по иерархии замечаний нет.

### Signature test
- Уникальное для продукта: эмодзи у трендов и их использование в подписях — по сути единственный узнаваемый ход. Структурно (карточки, таблица, список) — без подписи, как в любом аналитическом дашборде.
- **Вывод:** подпись есть только в контенте (эмодзи), в структуре и визуале — нет.

### Token test
- Токены: `--background`, `--foreground`, `--primary`, `--card`, `--muted`, `--border`, `--destructive`. Подходят любому SaaS, не привязаны к домену (генерация, тренды, снимки).
- **Вывод:** нейтральная палитра без «цветового мира» продукта.

## 3. Craft Foundations

### Глубина (Depth)
- **Проблема:** Карточки используют и `border`, и `shadow-sm` (Card: `rounded-lg border bg-card ... shadow-sm`). В гайде: выбрать один подход (только границы / только тени / только сдвиг поверхностей) и не смешивать.
- **Рекомендация:** Либо убрать `shadow-sm` и оставить только границы (логично для плотного инструмента), либо оставить тень и ослабить границу до `border-border/60` и не добавлять новые тени в других местах.

### Границы (Borders)
- Таблица: `border-b` у строк; карточки и списки: `border`, `border-border/60`. В целом границы не перегружают.
- В index.css `--border: 214.3 31.8% 91.4%` — сплошной светло-серый. Для «шёпота» можно ввести уровень с пониженной непрозрачностью (например, отдельный токен для мягкого разделения).

### Поверхности (Surface elevation)
- `--card` и `--background` в light оба `0 0% 100%` — карточки не приподняты тоном. Иерархия только за счёт границы и тени. Нет явной шкалы уровней (base, 1, 2, …).
- **Рекомендация:** Либо задать лёгкий сдвиг фона карточки (например, на пару процентов темнее/светлее фона), либо осознанно принять «одну плоскость + границы».

### Отступы и сетка
- В TrendsAnalyticsPage: `py-2.5` (10px) у элементов списка топа трендов и в UsersPage у похожих блоков. 10px не кратно 4.
- Остальное в основном по шкале Tailwind (4, 8, 12, 16, 24…).
- **Рекомендация:** Заменить `py-2.5` на `py-2` (8px) или `py-3` (12px), чтобы всё держалось на сетке 4px.

### Типографика
- Уровни: заголовок страницы (крупный, semibold), подзаголовок (text-sm, muted), заголовки карточек, тело таблицы/списка, мета (мелкий, muted). Явных четырёх уровней текста (primary / secondary / tertiary / muted) в токенах и компонентах нет.
- **Рекомендация:** Ввести и использовать четыре уровня текста (в т.ч. tertiary для подписей к ячейкам/метрик), чтобы иерархия была системной, а не только за счёт размера.

### Семантика цветов (успех / предупреждение / ошибка)
- В коде: `text-emerald-600`, `bg-emerald-500`, `text-red-600`, `text-amber-600` и т.п. напрямую. В CSS-переменных есть только `--destructive`.
- **Проблема:** Нет общих токенов для success и warning; сложнее менять тему и сохранять единый тон.
- **Рекомендация:** Добавить в `:root` и `.dark` токены, например `--success` и `--warning`, и использовать их (через Tailwind extend или классы), вместо жёстких emerald/amber/red.

## 4. Конкретные правки (приоритет)

| Приоритет | Что | Где |
|-----------|-----|-----|
| Высокий | Убрать `py-2.5`, заменить на `py-2` или `py-3` | TrendsAnalyticsPage (список топа), UsersPage (аналогичные блоки) |
| Высокий | Зафиксировать стратегию глубины: убрать `shadow-sm` у Card или ослабить border и не смешивать стили | `components/ui/card.tsx`, при необходимости глобальные стили |
| Средний | Ввести `--success` и `--warning`, использовать их вместо emerald/amber/red | `index.css`, tailwind.config, страницы с метриками и статусами |
| Средний | Описать четыре уровня текста и применить (primary, secondary, tertiary, muted) | Дизайн-токены / типографика в UI-компонентах и страницах |
| Низкий | Смягчить границы через токен с пониженной непрозрачностью | index.css, компоненты таблицы/карточек |
| Низкий | Вынести «подпись» продукта в структуру (например, блок «тренд + эмодзи + мини-превью» или акцент на эмодзи в таблице) | TrendsAnalyticsPage, при необходимости общие компоненты трендов |

## 5. system.md

В проекте нет `.interface-design/system.md`. После внедрения правок имеет смысл завести файл и записать:

- Стратегию глубины (только границы / только тени).
- Базовый шаг сетки (4px) и правило «только кратные значения».
- Токены семантики (success, warning) и четыре уровня текста.
- Паттерны карточки аналитики и строки тренда (отступы, границы, hover).
