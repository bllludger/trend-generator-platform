# Code review: метрики Prometheus / Grafana перед продом

## 1. Critical bugs

- **Нет.** Критических багов, которые ломают бизнес-логику или приводят к падению сервиса, не найдено. Использование `job` в `generation_v2` в блоке `except` защищено проверкой `if job and job.status in (...)` — `jobs_failed_total.inc()` вызывается только при существующем `job`.

---

## 2. High-risk issues

### 2.1 Блокировка `/metrics` из-за Redis (стабильность)

**Файл:** `app/utils/metrics.py`, `_update_celery_queue_gauges()`.

При каждом запросе к `/metrics` создаётся новое подключение к Redis (broker) и вызывается `r.llen(queue_name)` для двух очередей. Таймауты для `redis.Redis.from_url()` не заданы — при недоступности или тормозах Redis запрос к `/metrics` может висеть до TCP-таймаута (десятки секунд). Это:

- увеличивает время ответа API для скрапа Prometheus;
- может приводить к таймаутам скрапа и пометке инстанса как down;
- в худшем случае блокирует воркер/процесс, обслуживающий `/metrics`.

**Рекомендация:** задать `socket_connect_timeout` и `socket_timeout` (например, 2–3 с) при создании клиента Redis и/или обернуть вызов в `asyncio.wait_for`/отдельный поток с таймаутом, чтобы при недоступности Redis быстро выходить и не блокировать ответ (при ошибке уже сейчас метрики не падают — `except: pass`).

### 2.2 Ключи очередей Celery в Redis

**Файл:** `app/utils/metrics.py`, цикл по `("celery", "generation")`.

По умолчанию Kombu/Celery с Redis использует имя очереди как имя ключа списка; в проекте `broker_transport_options` / `global_keyprefix` не заданы, поэтому ключи `celery` и `generation` ожидаемо верны. Если в будущем добавится префикс или другой бэкенд, `celery_queue_length` будет показывать 0 или неверные значения. Сейчас — риск низкий, но стоит зафиксировать в конфиге/доках зависимость от «стандартных» имён ключей.

### 2.3 Исключения в middleware и при записи метрик

**Файл:** `app/middleware/metrics_middleware.py`.

Если `http_requests_total.labels(...).inc()` или `observe()` выбросят исключение (например, из-за невалидного значения метки в какой-то версии клиента), весь ответ пользователю упадёт. Сейчас `method` и `path` контролируются нами, риск мал. Для максимальной устойчивости можно обернуть запись метрик в `try/except` и логировать ошибку, не ломая ответ.

## 3. Logic problems

### 3.1 Двойной учёт при retry задачи Celery

**Файлы:** `app/workers/tasks/generate_take.py`, `app/workers/tasks/generation_v2.py`.

При повторном запуске одной и той же задачи (retry после сбоя воркера) снова вызываются:

- `takes_created_total.inc()` в начале `generate_take`;
- в `generation_v2` при успехе — `jobs_succeeded_total.inc()` и т.д.

В итоге один и тот же take/job может быть посчитан дважды (как созданный и как успешный/неуспешный). Для «количества обработанных задач» это ожидаемо; для «количества уникальных созданий» — завышение. Решение зависит от целей метрик: либо принять как есть (считать «запуски»), либо инкрементить «created» только при первом запуске (например, по idempotency key или по статусу take/job при входе в задачу).

### 3.2 Согласованность «delivered» / «failed» в deliver_hd

**Файл:** `app/workers/tasks/deliver_hd.py`.

Логика разветвлений проверена: при `status_chat_id` не None и успешной отправке инкрементируется только «delivered» внутри `try`; при ошибке отправки — «failed»; при отсутствии `status_chat_id` в конце инкрементируется «delivered». Двойного учёта и пропусков нет.

## 4. Edge cases missed

### 4.1 Health check и счётчик сбоев

**Файл:** `app/api/routes/health.py`.

Любая ошибка в readiness (БД, Redis, сетевая) приводит к `api_health_check_failures_total.inc()`. Кратковременный сбой (например, таймаут Redis на 1 с) тоже даёт инкремент. Для алертинга это нормально (мы хотим видеть факт недоступности), но стоит иметь в виду: метрика не различает «полный отказ» и «одноразовый таймаут».

### 4.2 Кардинальность метрик

- **product_events_track_total** по `event_name` и `status`: множество событий ограничено кодом (bot_started, pay_success, …), кардинальность приемлемая.
- **pay_success_total / pay_initiated_total** по `pack_id`: значения задаются приложением (pack.id, "unlock"), не пользователем — ок.
- **http_requests_total** по `path`: нормализация через `_normalize_path` (uuid, numeric id, long ids → placeholders) ограничивает кардинальность.

Дополнительно ограничивать не обязательно; при добавлении новых метрик с пользовательскими или неограниченными лейблами — учитывать кардинальность.

### 4.3 Отсутствие метрик для части сценариев

- В боте есть несколько веток успешной оплаты (Stars, YooMoney, unlock и т.д.); везде, где вызывается успешное завершение платежа, проверено наличие соответствующих `pay_success_total` и при необходимости `payment_amount_stars_total`. Пропущенных веток не обнаружено.
- При падении задачи до первого инкремента (например, падение при разборе аргументов) ни `takes_failed_total`, ни `jobs_failed_total` не увеличатся — будет только лог и повторная попытка Celery. Это приемлемо для мониторинга «обработанных» исходов.

---

## 5. What must be fixed before release

1. **Сделано:** снижен риск блокировки `/metrics` из-за Redis — в `_update_celery_queue_gauges()` для Redis-клиента заданы `socket_connect_timeout` и `socket_timeout` (2 с).

2. **Сделано:** в middleware метрик запись метрик обёрнута в `try/except`, при ошибке пишется лог и ответ не ломается.

3. Остальное (двойной учёт при retry, ключи Celery, кардинальность) оставлено как есть; при необходимости доработать в следующих итерациях.

---

## 6. Final verdict

**Ready** — обязательные правки внесены (таймауты Redis для `/metrics`, защита middleware от исключений при записи метрик). Фичу можно выкатывать в прод.
