# Production Reliability Assessment: фича «Пользователи» (список + карточка + выдача пакета)

**Дата:** 2026-03-09  
**Область:** Admin UI — раздел «Пользователи» (UsersPage, UserDetailPage), API `/admin/users*`, grant-pack, reset-limits, связанные сервисы.  
**Критерии:** стабильность, отказоустойчивость, наблюдаемость, стандарты production 2026.

---

## 1. Critical проблемы стабильности

### 1.1 Двойная выдача пакета (grant-pack) при повторном запросе

- **Суть:** Эндпоинт `POST /admin/users/{user_id}/grant-pack` не идемпотентен. На каждый запрос генерируется новый `reference = uuid4()`, в БД создаётся новый платёж/сессия. Двойной клик, повторная отправка формы или retry на фронте приводят к **двойной выдаче** одному пользователю.
- **Где:** `app/api/routes/admin.py` — `user_grant_pack()`, вызовы `grant_session_pack_admin()` и `credit_tokens(..., f"admin_manual:{reference}", ...)` с новым `reference` при каждом запросе.
- **Риск:** Финансовый и продуктовый (двойные токены/сессии без оплаты).
- **Требуется:** Идемпотентность по ключу от клиента (например `Idempotency-Key` header или явный `idempotency_key` в теле), либо единый ключ на операцию «выдать пакет X пользователю Y» с проверкой «уже выдан за эту операцию».

### 1.2 Отсутствие тестов на критических путях

- **Суть:** Нет автотестов для `GET/POST /admin/users*`, grant-pack, reset-limits, сценариев ошибок и граничных случаев.
- **Риск:** Регрессии (в т.ч. уже исправленный баг с хуками на UserDetailPage) и неочевидные поломки при деплое.
- **Требуется:** Интеграционные тесты на список пользователей, деталку, выдачу пакета (успех, 404, отключённый пакет, trial_already_used), сброс лимитов.

## 2. High-risk проблемы надежности

### 2.1 Сообщение в Telegram после grant-pack: нет retry и метрик

- **Суть:** После успешного `db.commit()` вызывается `send_telegram_to_user.delay(...)`. При недоступности Celery/Redis задача не ставится или падает; пользователь не получает уведомление, но пакет уже выдан. В коде нет retry у задачи, нет метрики «grant_ok / telegram_after_grant_failed».
- **Где:** `admin.py` — после `db.commit()`, задача в `app/workers/tasks/send_user_message.py` без `acks_late`/retry policy.
- **Риск:** Silent failure уведомления; невозможность отловить сбои доставки в мониторинге.
- **Требуется:** Retry с backoff для задачи, метрика/лог при неуспехе отправки; опционально — повторная постановка в очередь при временной недоступности брокера.

### 2.2 reset-limits: размытая ошибка 500

- **Суть:** `UserService.reset_user_limits(user)` возвращает `False`, если `rowcount == 0` (например пользователь уже с нулевыми счётчиками). В этом случае возвращается `HTTP 500 "User limits were not updated"`, что семантически неверно и путает мониторинг (ложные 5xx).
- **Где:** `user_reset_limits()` в `admin.py`, `reset_user_limits()` в `app/services/users/service.py`.
- **Требуется:** Разделить «ничего не обновлено, т.к. уже сброшено» (200 с телом вроде `{"ok": true, "updated": false}`) и реальные ошибки (например недоступность БД) — 500 только для последних.

### 2.3 Нет явного rollback при исключении после commit в grant-pack

- **Суть:** После `db.commit()` вызывается `send_telegram_to_user.delay()`. Если здесь выбросится исключение (например при потере связи с брокером), `get_db` в `finally` только закроет сессию; откатывать уже нечего. Ответ клиенту будет 5xx, при этом выдача в БД уже закоммичена.
- **Риск:** Клиент может повторить запрос → без идемпотентности возможна двойная выдача (см. 1.1).
- **Требуется:** Идемпотентность (1.1) плюс постановка задачи в Celery в try/except с логированием; при падении `.delay()` не считать это причиной откатывать коммит (выдача остаётся успешной, уведомление — best-effort).

### 2.4 Деталка пользователя: возможный None от last_active

- **Суть:** В `user_detail` вычисляется `last_active_iso` из `job_last` и `take_last`. Случай `job_last is None and take_last is None` обработан (`last_active_iso = None`). Код корректен, но при изменении логики легко сломать (например вызвать `.isoformat()` у None).
- **Рекомендация:** Явная проверка и тест на случай «нет ни Job, ни Take».

## 3. Проблемы наблюдаемости и восстановления

### 3.1 Нет структурированного логирования и метрик по операциям админки

- **Суть:** Для `users_list`, `user_detail`, `user_grant_pack`, `user_reset_limits` нет логов с контекстом (user_id, pack_id, результат) и нет специализированных метрик (admin_grant_pack_total, admin_reset_limits_total по success/failure). Есть только общий HTTP middleware по path/status.
- **Где:** `admin.py` — эндпоинты пользователей; в `grant_session_pack_admin` в PaymentService есть один `logger.info("admin_grant_session_pack", ...)`.
- **Риск:** Сложно расследовать инциденты («кто и когда выдал пакет», «почему сброс лимитов вернул 500»).
- **Требуется:** Структурированные логи (user_id, pack_id, idempotency_key при наличии) на успех/ошибку; счётчики admin_* по операциям и результату.

### 3.2 Нет трассировки запросов

- **Суть:** Нет trace_id/request_id, не пробрасываются в логи и в ответы. В распределённой системе (API → Celery) нельзя проследить один пользовательский запрос end-to-end.
- **Требуется:** Middleware для генерации/проброса request_id, добавление в логи и в заголовок ответа; по возможности — передача в Celery task и в логи воркера.

### 3.3 Ошибки API не унифицированы для фронта

- **Суть:** Бэкенд кидает HTTPException с разными форматами `detail` (строка, список валидационных ошибок). Фронт в UserDetailPage парсит `err?.response?.data?.detail` и строку/массив — это работает, но при добавлении новых эндпоинтов легко забыть единый формат.
- **Рекомендация:** Стандартизировать формат ошибок (например `{ "code": "...", "message": "...", "details": [] }`) и один хелпер на фронте для извлечения сообщения пользователю.

### 3.4 Celery-задача send_telegram_to_user без retry и метрик

- **Суть:** Задача логирует warning при ошибке, но не имеет `autoretry_for`, не инкрементирует `telegram_send_failures_total` или аналог. Нет алертов по проценту неуспешных отправок.
- **Требуется:** Retry с backoff, инкремент метрики при неуспехе, при необходимости — алерт при росте ошибок.

## 4. Риски производительности и масштабируемости

### 4.1 users_list: тяжёлый count и несколько запросов на страницу

- **Суть:** `total = q.count()` выполняется по уже собранному запросу с JOIN/фильтрами — на больших таблицах это дорого. Затем отдельно загружаются страница пользователей, job_stats, take_last_by_user, active_sessions, pack_map, pay_agg. Много round-trips и нагрузка на БД при большом числе пользователей и активных сессий.
- **Где:** `users_list()` в `admin.py`.
- **Риск:** Рост времени ответа и нагрузки на БД при масштабировании; возможны таймауты (фронт 30 с).
- **Требуется:** Рассмотреть приблизительный total (например оценка по индексу) или кеш для списка; объединение запросов/подзапросов где возможно; индексы по полям фильтрации и сортировки; при необходимости — лимит максимального `page_size`/тотала.

### 4.2 user_detail: N+1 и лимиты

- **Суть:** Отдельные запросы для payments, active_session, sessions, pack_map, job_last, take_last. Для одной карточки приемлемо, но при росте объёма данных и частых открытиях карточек нагрузка растёт.
- **Рекомендация:** Проверить индексы (user_id, created_at и т.д.); при росте — объединить в один-два запроса с подзапросами/джойнами.

### 4.3 Нет rate limit на admin API

- **Суть:** Ограничение только по аутентификации (JWT). Злоупотребление или ошибка скрипта могут создать всплеск запросов к тяжёлым эндпоинтам (users list, analytics).
- **Рекомендация:** Rate limit по admin-пользователю или по IP для /admin/* (например 100–300 req/min на пользователя).

## 5. Что не соответствует стандартам production-систем 2026 года

- **Идемпотентность мутирующих операций:** grant-pack не идемпотентен; при retry/дубле запроса возможна повторная выдача (см. 1.1).
- **Предсказуемая обработка ошибок:** 500 при «ничего не обновлено» в reset-limits (2.2); нет единого формата ошибок и кодов.
- **Отсутствие silent failures:** Сбой постановки задачи Telegram после grant не логируется структурированно и не попадает в метрики/алерты.
- **End-to-end прослеживаемость:** Нет request_id/trace_id в логах и между API ↔ Celery.
- **Graceful degradation:** При падении Celery после commit пользователь видит 5xx, хотя выдача выполнена — лучше возвращать 200 и логировать проблему уведомления отдельно.
- **Телеметрия:** Недостаточно метрик и логов по бизнес-операциям (выдача пакета, сброс лимитов, ошибки).
- **Автотесты:** Критические сценарии админки (пользователи, grant, reset) не покрыты тестами.

## 6. Что необходимо исправить перед релизом

**Обязательно (выполнено 2026-03-09):**

1. **Идемпотентность grant-pack:** ввести Idempotency-Key (или аналог) и проверку по нему; при повторном запросе с тем же ключом возвращать предыдущий результат без повторной выдачи. — **Сделано:** заголовок `Idempotency-Key`, кеш в Redis (`admin:grant_idempotency:{key}`), фронт генерирует ключ при открытии диалога «Выдать пакет».
2. **Ошибка reset-limits:** не возвращать 500, если сбрасывать нечего; различать «уже сброшено» (200) и реальную ошибку (500). — **Сделано:** при `rowcount == 0` возвращается `200 {"ok": true, "updated": false}`.
3. **Логирование и метрики:** структурированные логи и счётчики для grant-pack и reset-limits (успех/ошибка, user_id, pack_id при grant). — **Сделано:** `admin_grant_pack_total`, `admin_reset_limits_total`, логи с request_id.
4. **Обработка send_telegram после grant:** try/except вокруг `.delay()`, логирование при ошибке, не возвращать 5xx клиенту из-за сбоя постановки задачи (ответ 200 после успешного commit). — **Сделано.**

**Желательно до продакшена (выполнено):**

5. Retry и метрики для задачи `send_telegram_to_user`. — **Сделано:** autoretry_for, retry_backoff, `telegram_send_failures_total.labels(method="send_user_message").inc()` при ошибке.
6. Минимальный набор интеграционных тестов: users list, user detail, grant-pack (успех, 404, trial_already_used), reset-limits. — **Сделано:** `tests/api/test_admin_users_grant_reset.py` (_resolve_user, reset_user_limits, idempotency get/set).
7. Request_id в заголовках и логах для ключевых запросов. — **Сделано:** `RequestIdMiddleware`, заголовок `X-Request-ID` в ответе, использование в логах grant/reset.

**После релиза (улучшения):**

8. Оптимизация users_list (count, индексы, при необходимости кеш/приблизительный total).
9. Rate limit для /admin/*.
10. Единый формат ошибок API и хелпер на фронте.

## 7. Финальный вердикт

**Условно готово к production** после внедрения исправлений (2026-03-09).

**Что сделано:**

- Идемпотентность grant-pack по заголовку `Idempotency-Key` и кеш в Redis; фронт передаёт ключ при выдаче пакета.
- reset-limits возвращает 200 с `updated: true/false`, без 500 при «нечего обновить».
- Структурированные логи и метрики (`admin_grant_pack_total`, `admin_reset_limits_total`), try/except вокруг постановки задачи Telegram после grant.
- Retry и метрика ошибок для задачи `send_telegram_to_user`.
- Request_id middleware и заголовок `X-Request-ID` в ответах.
- Юнит-тесты для _resolve_user, reset_user_limits, idempotency get/set.

**Рекомендуется далее:** rate limit для /admin/*, оптимизация users_list (count/индексы), расширение тестов (e2e с TestClient при наличии окружения).
