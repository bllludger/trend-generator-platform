# Code review: фича «Оплата переводом» (админка + телеметрия)

Проверены: `GET /admin/bank-transfer/pay-initiated`, расширение `receipt-logs` (expected_rub, date_from, date_to), фильтр дат в `GET /admin/payments`, запись `pay_success` в PaymentService, фронт BankTransferPage (блоки Инициации, лог чеков, успешные платежи).

---

## 1. Critical bugs

### 1.1 Нет валидации path при отдаче файла чека (path traversal)

**Файл:** `app/api/routes/admin.py`, `bank_transfer_receipt_log_file`.

Используется `row.file_path` из БД и `os.path.join(base, path)`. Если в БД когда-либо окажется значение вроде `../../../etc/passwd` (ручное изменение, миграция, баг), итоговый путь может выйти за пределы `storage_base_path` и отдать произвольный файл.

**Нужно:** после сборки полного пути проверять, что канонический путь лежит внутри `storage_base_path` (например `os.path.realpath(path).startswith(os.path.realpath(base))`), и иначе возвращать 404.

### 1.2 Commit внутри track() в общей сессии бота

**Файл:** `app/services/product_analytics/service.py` — `track()` вызывает `self.db.commit()`.

В боте: `with get_db_session() as db:` → `process_session_purchase_bank_transfer(db)` → внутри вызывается `ProductAnalyticsService(db).track(...)`, который делает `self.db.commit()`. В итоге коммитится вся текущая транзакция (payment + session), а не только product_event. Дальше бот делает `_create_receipt_log(db, ...)` и при выходе из `with` — ещё один `db.commit()` (уже только receipt_log).

**Вывод:** поведение согласованное: платёж и сессия коммитятся в track(), лог чека — при выходе из контекста. Критического бага нет, но архитектурно неочевидно: один вызов track() коммитит чужую транзакцию. Для прода это не блокер, но стоит в бэклоге рассмотреть вынос телеметрии в отдельную сессию или отложенную запись.

## 2. High-risk issues

### 2.1 Path traversal при отдаче файла чека (дубль 1.1)

Описано выше — обязательно исправить до прода: проверка, что результирующий путь находится под `storage_base_path`.

### 2.2 Нет обработки ошибок API на фронте

**Файл:** `admin-frontend/src/pages/BankTransferPage.tsx`.

Для `payInitiatedData`, `receiptLogsData`, `btPaymentsData` не используются `isError` / `error` из `useQuery`. При 500 или сетевой ошибке пользователь видит «Записей нет» вместо сообщения об ошибке и может решить, что данных нет.

**Нужно:** показывать состояние ошибки (тост или блок «Не удалось загрузить данные») и при необходимости кнопку повтора.

### 2.3 Пагинация не сбрасывается при смене фильтров

При смене дат или суммы страница (например, 5) сохраняется. Если по новым фильтрам страниц меньше, получаем пустой список без пояснения.

**Нужно:** при изменении `date_from`, `date_to`, `price_rub`, `expected_rub` и т.п. сбрасывать `payInitiatedPage`, `receiptLogPage`, `btPaymentsPage` в 1 (через `useEffect` по зависимостям фильтров).

### 2.4 date_to < date_from не обрабатывается

Бэкенд и фронт не проверяют, что `date_to >= date_from`. При датах «наоборот» бэкенд возвращает пустой список, фронт показывает «Записей нет» без подсказки.

**Рекомендация:** на бэкенде при `date_to < date_from` возвращать 400 с ясным сообщением; на фронте опционально подсвечивать невалидный диапазон или подставлять корректные даты.

## 3. Logic problems

### 3.1 pay_success только для session pack

**Файл:** `app/services/payments/service.py`.

`track("pay_success", ...)` вызывается только в `process_session_purchase_bank_transfer`. Платежи через `credit_tokens_manual` (standalone-пакеты, не из лестницы) не дают события `pay_success` в product_events. Воронка и отчёты по выручке для банковского перевода будут учитывать только «сессионные» пакеты.

**Рекомендация:** в бэкенде при успешном `credit_tokens_manual` для bank_transfer тоже вызывать `track("pay_success", ...)` с теми же свойствами (method, price_rub и т.д.), если нужна полная картина по всем переводам.

### 3.2 Фильтр по telegram_user_id без trim на бэкенде

**Файл:** `app/api/routes/admin.py`, `bank_transfer_pay_initiated` и `bank_transfer_receipt_logs`.

Параметр `telegram_user_id` не обрезается. Значение `" 123456"` или `"123456 "` не совпадёт с `"123456"` в БД.

**Рекомендация:** делать `telegram_user_id = telegram_user_id.strip() if telegram_user_id else None` перед использованием (и не добавлять в запрос при пустой строке после strip).

### 3.3 JSONB price_rub: возможное несовпадение типов

**Файл:** `app/api/routes/admin.py`, строка ~1369.

Используется `cast(ProductEvent.properties["price_rub"].astext, SqlNumeric(12, 2)) == price_rub`. В БД в JSON может лежать число (129 или 129.0). В PostgreSQL `->>'price_rub'` даёт текст. Если в старых событиях поле отсутствует или не число, выражение может дать NULL и строка отфильтруется. Для текущего контракта (числа 129, 199) поведение приемлемо, но при добавлении других сумм стоит убедиться, что везде один формат (например, всегда число в JSON).

## 4. Edge cases missed

### 4.1 Пустые даты и «только одна дата»

Если задать только `date_from` или только `date_to`, бэкенд отрабатывает корректно (односторонний предел). На фронте пустые поля не отправляются — ок. Явная документация в UI («Период: обе даты опциональны») снизит путаницу.

### 4.2 Очень большая страница

`page_size` ограничен 200. При `page_size=200` и большом `total` запросы к БД и объём JSON вырастут. Для админки приемлемо; при росте нагрузки можно ужесточить лимит (например, 50) или добавить предупреждение.

### 4.3 Удалённый пользователь и pay-initiated

При фильтре по `telegram_user_id` и удалении пользователя (CASCADE по product_events) записей не будет. Без фильтра события удалённых пользователей уже не показываются из-за CASCADE. Отдельная обработка не требуется.

### 4.4 Ошибка при запросе файла чека

`handleOpenReceiptFile` при ошибке показывает toast. При 404 или 403 пользователь видит общее сообщение — достаточно для первого релиза; при желании можно различать 404 и «нет доступа».

### 4.5 refetchInterval 20–30 сек

Три блока с интервалом 20–30 сек дают постоянный поток запросов при открытой вкладке. Для небольшого числа админов допустимо; при росте — рассмотреть увеличение интервала или отключение авто-обновления по умолчанию.

## 5. What must be fixed before release

Обязательно:

1. **Path traversal в `bank_transfer_receipt_log_file`:** проверять, что результирующий путь находится внутри `storage_base_path` (через `os.path.realpath` и префикс базового каталога), иначе 404.
2. **Обработка ошибок запросов на фронте:** для трёх useQuery (pay-initiated, receipt-logs, bank-transfer payments) использовать `isError`/`error`, показывать сообщение об ошибке и не отображать «Записей нет» при падении запроса.
3. **Сброс страницы при смене фильтров:** при изменении дат/суммы/Telegram ID сбрасывать номер страницы в 1 для соответствующих таблиц.

Желательно до прода:

4. Валидация `date_to >= date_from` (400 на бэкенде или подсказка на фронте).
5. Trim `telegram_user_id` на бэкенде в обоих эндпоинтах.

В бэклоге:

6. Вызов `track("pay_success", ...)` для банковских переводов через `credit_tokens_manual`.
7. Рефакторинг телеметрии: не коммитить общую сессию из `track()` (отдельная сессия или очередь событий).

---

## 6. Final verdict

**Not ready** для прода в текущем виде.

Критично закрыть: path traversal при отдаче файла чека. После этого и исправления пунктов из раздела 5 (обязательные) фичу можно выкатывать. Остальное (обработка ошибок, сброс пагинации, валидация дат, trim) заметно улучшит устойчивость и UX и должно быть сделано до или сразу после релиза.
