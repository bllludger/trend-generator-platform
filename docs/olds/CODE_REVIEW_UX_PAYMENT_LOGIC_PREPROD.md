# Code review: UX и логика оплаты (перед продом)

**Область:** сброс тренда, счётчик 9/10, баннер после 4K, «Оплата подтверждена» без кнопок, скрытие оплаты при активном пакете.

---

## 1. Critical bugs

### 1.1 Нет экранирования Markdown в пользовательских строках

**Файлы:** `app/bot/main.py` (pack_done_, pack_check), `app/api/routes/webhooks.py` (_send_pack_success_message).

Сообщения отправляются с `parse_mode="Markdown"`. В текст подставляются `pack_name` и `pack_emoji` из БД без экранирования. Если в названии пакета есть `_`, `*`, `[`, `` ` `` и т.д., Telegram может сломать разметку (курсив, ссылки) или вернуть ошибку при отправке.

**Рекомендация:** экранировать символы Markdown в `pack_name` и `pack_emoji` (или использовать `parse_mode="HTML"` и экранировать `<`, `>`, `&`) перед подстановкой в строку. Либо не использовать parse_mode для этого сообщения и форматировать только фиксированные части (например, жирный только у «Оплата подтверждена»).

### 1.2 «Ошибка активации» без пояснения и действий

**Файл:** `app/bot/main.py`, обработчик `pack_check_callback`.

При `payment_status == "succeeded"` вызывается `process_session_purchase_yookassa_link`. Если вернулись `payment_obj is None` или `session is None`, пользователю показывается только `callback.answer("Ошибка активации", show_alert=True)`. В чат не отправляется ни сообщение, ни кнопка «Помощь»/«В меню». Пользователь не понимает, что делать дальше.

**Рекомендация:** после `await callback.answer("Ошибка активации", show_alert=True)` отправить в чат сообщение с просьбой обратиться в поддержку и кнопкой «Помощь»/«В меню».

## 2. High-risk issues

### 2.1 Регрессия: после оплаты пакета больше не напоминаем про «забрать последнее 4K»

**Файл:** `app/api/routes/webhooks.py`.

Раньше в «Оплата подтверждена» добавлялись текст «Можно забрать последнее фото в 4K» и кнопка «Открыть фото в 4K», если у пользователя был pending HD. Сейчас сообщение только текстовое и без кнопок; про уже выбранный вариант и возможность забрать 4K не говорится. Пользователь, который оплатил пакет и до этого уже выбрал вариант (но не нажал «Открыть в 4K»), может не понять, что нужно вернуться к предыдущему сообщению и нажать доставку 4K.

**Рекомендация:** в текст после активации пакета добавить условный блок: если есть pending 4K — одна строка вида «Вы уже выбрали снимок — вернитесь к сообщению с кнопкой «Открыть фото в 4K» и нажмите её.» Без кнопок, только текст.

### 2.2 Гонка: webhook «Оплата подтверждена» приходит раньше превью

Пользователь оплачивает по ссылке и сразу возвращается в бота по `start=pack_done_XXX` или получает сообщение от webhook. Сообщение говорит: «Вернитесь выше в чате и выберите лучший снимок». Если воркер с генерацией ещё не отправил три варианта, «выше» пусто — инструкция вводит в заблуждение.

**Рекомендация:** принять как допустимый edge case и при необходимости смягчить формулировку (например: «Когда появятся три варианта снимка, выберите лучший — тогда получите его в полном качестве.»).

### 2.3 Воркер показывает «Выбрать и оплатить», бот при клике даёт 4K без оплаты

**Файл:** `app/workers/tasks/generate_take.py`.

Логика «есть ли у пользователя активный платный пакет с остатком» в воркере считается только по **сессии этого take** (`take.session_id`). Если пользователь во время генерации купил пакет (новая сессия), take всё ещё привязан к старой (например, free_preview). Воркер после commit отправляет текст с «Выбрать и оплатить» и кнопками «Выбрать и оплатить A/B/C». В боте в `choose_variant` проверяется `get_active_session(user.id)` (уже новая сессия с остатком), поэтому показ оплаты подавляется и пользователь получает 4K из пакета. Итог: сообщение призывает оплатить, по факту оплаты нет — путаница и недоверие.

**Рекомендация:** в воркере при выборе текста/кнопок учитывать не только сессию take, но и наличие у пользователя (`take.user_id`) любой активной платной сессии с остатком (например, через отдельный запрос по `user_id` к активным сессиям). Если такая сессия есть — использовать вариант без «оплатить» и с формулировкой про пакет.

## 3. Logic problems

### 3.1 «Доступно фото: 0 из 0» при отсутствии user/session

**Файлы:** `app/bot/main.py` (pack_done_, pack_check при status == "completed").

Если по какой-то причине `get_by_telegram_id(telegram_id)` вернул `None` или у пользователя нет активной сессии, в сообщение подставляется `remaining = 0`, `takes_limit = 0`. Получается «Доступно фото: 0 из 0», что бессмысленно и выглядит как ошибка.

**Рекомендация:** если `user is None` или `session is None` в этом сценарии — не показывать блок «Доступно фото: N из M», а написать нейтрально: «Пакет активирован.» или «Пакет … активирован. Вернитесь в чат и выберите лучший снимок.»

### 3.2 pack_check: при уже completed не делаем commit после ответа

В ветке `pack_order.status == "completed"` мы только читаем данные и шлём сообщение. Изменений в БД нет, commit не требуется — поведение корректно. Отдельная проверка: при только что выполненном `mark_completed(pack_order.id)` commit происходит при выходе из `with get_db_session()`, то есть после `await callback.message.answer(...)`. Порядок корректен.

## 4. Edge cases missed

### 4.1 Deep link pack_done_ без «трёх вариантов выше»

Пользователь оплатил в браузере и перешёл по ссылке в бота. В чате нет предыдущих сообщений с вариантами. Текст «выберите лучший из трёх вариантов выше» и «вернитесь выше в чате» не соответствуют ситуации.

**Рекомендация:** оставить как есть или слегка обобщить формулировку (см. п. 2.2), не добавляя сложной логики «есть ли выше варианты».

### 4.2 deliver_hd: не только free_preview попадает в else

**Файл:** `app/workers/tasks/deliver_hd.py`, `_try_upsell_after_hd`.

В `else` попадают все сессии с `pack_id != "free_preview"` == False, т.е. все, у кого `pack_id` в нижнем регистре равен `"free_preview"`. Любой другой pack_id (включая пустую строку, null, legacy-значения) попадает в `is_paid` и не в этот return. То есть мы не показываем баннер только для явного free_preview. Для «неизвестного» pack_id баннер не показывается только если они попали в `is_paid` (что не так для пустого pack_id). Проверка `(session.pack_id or "").strip().lower() == "free_preview"` корректна: только free_preview не получает баннер; остальные получают баннер в ветке is_paid. Пустой или None pack_id даёт is_free_preview=False, is_paid=True — тогда показывается платный баннер (остаток + «Купить ещё 4K»). Для таких сессий это может быть неверно, но это уже общая логика, не регрессия данной фичи.

### 4.3 Долгое удержание сессии БД в pack_check_callback

В `pack_check_callback` один `with get_db_session()` охватывает несколько `await` (YooKassa, ответы в Telegram). Соединение держится открытым на время сетевых вызовов. Это существующий паттерн в боте; для продакшена стоит со временем вынести тяжёлые/внешние вызовы за пределы контекста сессии или использовать отдельную сессию только для быстрых операций с БД. В рамках этой фичи не менялось.

## 5. What must be fixed before release

Обязательно перед выкатом:

1. **Markdown:** экранировать `pack_name` и `pack_emoji` (или перейти на HTML и экранировать там) во всех сообщениях «Оплата подтверждена» (бот: pack_done_, pack_check; webhook: _send_pack_success_message).
2. **Ошибка активации:** при `not (payment_obj and session)` после успешного платежа в pack_check не только показывать alert «Ошибка активации», но и отправить в чат сообщение с кнопкой «Помощь»/«В меню».

Желательно до/вскоре после релиза:

3. **Воркер и активный пакет:** при формировании сообщения после генерации проверять по `take.user_id` наличие любой активной платной сессии с остатком, а не только сессию take; при наличии показывать вариант текста/кнопок без «оплатить».
4. **«0 из 0»:** при отсутствии user или session в сценарии «Оплата подтверждена» не выводить «Доступно фото: 0 из 0», заменить на короткий нейтральный текст.
5. **Pending 4K после оплаты пакета:** в тексте webhook/бота после активации пакета при наличии pending 4K добавить одну фразу-напоминание (без кнопок), что можно вернуться к кнопке «Открыть фото в 4K».

---

## 6. Final verdict

**Not ready** для прода без правок.

Критично: возможный падение/некорректное отображение из-за Markdown и тупиковый сценарий «Ошибка активации» без сообщения в чат. После исправления п. 1 и п. 2 из раздела 5 фичу можно выкатывать с принятием риска по п. 2.2 и 2.3 (текст «выше» и рассинхрон воркер/бот), с планом доработок по п. 3–5.
