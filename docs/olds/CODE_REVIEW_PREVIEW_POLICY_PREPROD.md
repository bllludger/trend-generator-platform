# Code review: политика превью (жёсткий ревью перед продом)

Проверены: баги, edge cases, регрессии, логика, стабильность, состояния, API, async, error handling.

---

## 1. Critical bugs

### 1.1 Утечка файловых дескрипторов в `send_media_group` при ошибке

**Файл:** `app/services/telegram/client.py`, метод `send_media_group`.

**Проблема:** Для каждого элемента альбома открывается файл (`open(media_path, "rb")`), затем вызывается `_api_call("sendMediaGroup", ...)`. Закрытие хендлов выполняется только после успешного `_api_call`. При любой ошибке (сеть, 429, неверный ответ) управление переходит в `except`, хендлы не закрываются — **утечка 3 файловых дескрипторов на каждый неуспешный альбом из 3 превью**.

**Сценарий:** Пользователь получает 3 варианта Take → воркер вызывает `send_media_group` → Telegram возвращает 429 или таймаут → исключение → 3 FD не закрыты. При повторных ошибках (нагрузка, лимиты) процесс может исчерпать лимит открытых файлов и падать или перестать отправлять фото.

**Исправление:** Открывать файлы в цикле и закрывать их в `finally` (или использовать контекстный менеджер / единый try/finally по всем открытым файлам), чтобы при любом исходе все хендлы были закрыты.

## 2. High-risk issues

### 2.1 Нет валидации `watermark_opacity` (0–255)

**Файл:** `app/services/app_settings/settings_service.py`, метод `update`.

**Проблема:** При обновлении принимается `row.watermark_opacity = int(data["watermark_opacity"])` без ограничения диапазона. В `apply_watermark_v2` значение попадает в альфа-канал RGBA. Значения вне 0–255 (например 300 или -1) могут приводить к неопределённому поведению PIL или к некорректному отображению вотермарка.

**Исправление:** Ограничить при сохранении:  
`row.watermark_opacity = max(0, min(255, int(data["watermark_opacity"])))`.

### 2.2 Регрессия при ошибке в `build_preview` внутри `prepare_delivery`

**Файлы:** `app/paywall/delivery.py`, `app/workers/tasks/generation_v2.py`.

**Проблема:** Если `build_preview` выбросит исключение (битый/нечитаемый файл, нехватка памяти, диск заполнен), `prepare_delivery` пробрасывает его дальше. В `generation_v2` ловится общий `Exception`, job переводится в FAILED, пользователь получает сообщение об ошибке — **сценарий обработан**. Но на диске уже мог быть создан `original_path` (после `shutil.copy2`), а превью не создано — остаётся «сиротский» файл. Для продакшена приемлемо, но при большом числе сбоев стоит учитывать очистку таких файлов (например, в cleanup-сервисе по маске `*_original.*` без пары `*_preview.*`).

**Рекомендация:** Не блокирует релиз; при желании добавить в cleanup удаление orphan original-файлов, если нет соответствующего preview.

## 3. Logic problems

### 3.1 Уже исправлено в текущей ветке

- **MIME при отправке в Telegram:** в `send_photo` и `send_media_group` используется `_mime_for_photo_path(path)` — MIME определяется по расширению (webp/jpeg/png). Регрессии нет.
- **Единый источник текста вотермарка:** в `get_watermark_params()` при пустом значении в БД используется `app_config.watermark_text`; в `preview_policy_get` для `watermark_text_effective` — тот же fallback через `getattr(app_settings, "watermark_text", ...)`. Логика совпадает.
- **Защита от некорректного max_dim:** в `get_take_preview_max_dim()` и `get_job_preview_max_dim()` значение ограничено `[PREVIEW_MAX_DIM_MIN, PREVIEW_MAX_DIM_MAX]` (100–4096). В API при PUT также валидация. Дополнительная защита в сервисе есть.
- **Валидация preview_format:** при PUT допускаются только `webp` и `jpeg`; иначе 400. В рантайме `get_preview_format()` возвращает только `webp` или `jpeg`.
- **Проверка original_path в build_preview:** в начале `build_preview` выполняется `os.path.isfile(original_path)` и при отсутствии файла выбрасывается `FileNotFoundError` с понятным сообщением.

### 3.2 Инициализация app_settings при старте

В `app/main.py` в lifespan вызывается `AppSettingsService(db).get_or_create()`, чтобы строка с id=1 существовала. Воркеры Celery не используют lifespan API, но при первом вызове `get_or_create()` в воркере строка будет создана при отсутствии. Конкурентный вызов `get_or_create()` из нескольких потоков в `generate_take` теоретически возможен; из-за id=1 одна вставка успешна, вторая может дать конфликт — риск низкий, при необходимости можно один раз прочитать настройки в главном потоке и передавать параметры в воркеры без доступа к БД.

## 4. Edge cases missed

### 4.1 Повреждённое изображение от провайдера

Если провайдер вернул битый или не-изображение, при открытии в `build_preview` (PIL) или в `apply_watermark_v2` может выброситься исключение. Оно пробрасывается в вызывающий код: в `generate_take` вариант помечается как failed, в `generation_v2` job переводится в FAILED. Пользователь видит ошибку. Edge case покрыт.

### 4.2 Старые превью в .png

Уже созданные Take/Job могут хранить пути к превью в .png. Новые — в .webp/.jpg. Код везде использует путь из БД и отправляет файл по этому пути; MIME берётся по расширению. Регрессий не ожидается.

### 4.3 Пустой payload в PUT /settings/preview-policy

При `PUT` с пустым телом или без разрешённых полей `data` пустой, `update` не вызывается, возвращается `preview_policy_get(db)` — текущие настройки. Поведение корректное.

### 4.4 watermarked.save() в build_preview

После `apply_watermark_v2` вызывается `watermarked.save(final_preview_path, "WEBP"|"JPEG", quality=...)`. При нехватке места на диске возможно `OSError`. Исключение пробрасывается, tmp-файл удаляется в `finally`. Воркер/задача обрабатывают ошибку (failed variant или FAILED job). Дополнительно можно логировать место на диске при ошибках записи.

## 5. What must be fixed before release

**Обязательно:**

1. **Утечка FD в send_media_group** — в `app/services/telegram/client.py` в методе `send_media_group` гарантировать закрытие всех открытых файлов при любом исходе (успех или исключение), например через `try`/`finally` с перебором и закрытием всех хендлов из `files`.

**Желательно перед продом:**

2. **watermark_opacity 0–255** — в `AppSettingsService.update()` при установке `watermark_opacity` ограничивать значение:  
   `row.watermark_opacity = max(0, min(255, int(data["watermark_opacity"])))`.

**После релиза (по желанию):**

3. Очистка orphan original-файлов в cleanup (если есть preview-политика и частые сбои на этапе build_preview).
4. Кэширование/передача параметров превью из главного потока в воркеры без доступа к БД в потоках.

## 6. Final verdict

**Ready** для продакшена после внесённых исправлений.

Исправлено в коде:
- Утечка FD в `send_media_group`: закрытие всех открытых файлов перенесено в `finally`, чтобы хендлы закрывались при любом исходе (успех или исключение).
- Валидация `watermark_opacity`: при сохранении ограничение `max(0, min(255, ...))`.

**Дополнительные правки стабильности (повторная проверка):**

- **Один успешный вариант Take:** Telegram API требует в `sendMediaGroup` от 2 до 10 элементов. При одном успешном варианте (два упали) вызываем `send_photo` вместо `send_media_group`, чтобы не получать 400 от Telegram.
- **prepare_delivery:** превью строится от `original_path` (копия raw), а не от `raw_file_path` — не зависим от возможного удаления/недоступности raw после копирования.
- **AppSettingsService.get_or_create:** при конкурентной вставке (два потока/воркера при первом запуске) обрабатывается `IntegrityError` — rollback и повторный `get()`, без падения.
- **send_media_group:** явная проверка 2–10 элементов до открытия файлов; при 0 или 1 элементе — понятная ошибка вместо 400 от API.

Рекомендуется после выката: мониторинг отправки превью и логов воркеров; при необходимости — очистка orphan original-файлов в cleanup.
