## 1. Critical bugs

### 1.1 Нет верхней границы для window_days / window_hours в телеметрии

**Файл:** `app/api/routes/admin.py`

- `telemetry_history(window_days=Query(7, ge=1))` — нет `le=...`. Запрос с `window_days=3650` даёт цикл по 3650 дням и тяжёлые запросы к БД.
- `telemetry_dashboard(window_hours=Query(24, ge=1))` — нет верхней границы; `window_hours=87600` (10 лет) даёт огромный since и тяжёлые агрегации.

**Риск:** DoS через админ-API или случайный большой параметр.

**Рекомендация:** задать разумные лимиты (исправлено в коде):
- `telemetry_history`: `window_days = Query(7, ge=1, le=90)`;
- `telemetry_dashboard`: `window_hours = Query(24, ge=1, le=720)`;
- `telemetry_product_metrics`: `window_days = Query(7, ge=1, le=90)`.

---

### 1.2 rollback() в ProductAnalyticsService.track() откатывает всю сессию

**Файл:** `app/services/product_analytics/service.py`

При исключении в `track()` вызывается `self.db.rollback()`. Если вызывающий код передаёт одну и ту же сессию и до этого делал в ней `db.add(...)` или менял объекты, rollback откатит и их.

**Текущее использование:** в боте под каждый блок телеметрии открывается отдельный `with get_db_session() as db`, в нём только вызов(ы) `track()`. В воркерах сессия общая на задачу; после `track()` обычно идёт только чтение или уже закоммиченные изменения (audit.log делает свой commit). То есть типичный сценарий не ломается, но контракт «track() может откатить переданную сессию» опасен при будущем рефакторинге.

**Рекомендация:** в доке или в коде явно зафиксировать: «передавать в track() отдельную сессию, не смешивать с другими изменениями в одной транзакции». Либо выделить для телеметрии отдельную сессию внутри `track()` (создать, add, commit, close), чтобы не трогать переданную.

## 2. High-risk issues

### 2.1 Жёстко заданные имена таблиц в literal_column

**Файл:** `app/api/routes/admin.py`, `_utc_date_job()` / `_utc_date_take()`:

```python
return literal_column("(jobs.created_at AT TIME ZONE 'UTC')::date")
return literal_column("(takes.created_at AT TIME ZONE 'UTC')::date")
```

Если переименовать таблицу в моделях (например, `Job.__tablename__`), запросы сломаются.

**Рекомендация:** константы или `Job.__table__.name` / `Take.__table__.name` при формировании строки для literal_column, либо короткий комментарий в коде, что при смене имени таблицы нужно править и здесь.

---

### 2.2 buy_pack (legacy): pack_stars может быть None

**Файл:** `app/bot/main.py`, `buy_pack`

Используются `pack_stars` и `round((pack_stars or 0) * rate, 2)` для телеметрии — это безопасно. Но в `send_invoice` передаётся `amount=pack_stars` без проверки. Если у пакета не задан `stars_price`, в Telegram уйдёт некорректный запрос.

**Рекомендация:** перед отправкой счёта проверять `if not pack_stars and pack_stars != 0: return` (или явно требовать заполненный `stars_price` для пакетов, доступных в buy:), чтобы не слать некорректный invoice.

---

### 2.3 Двойная сессия и возможная гонка в buy_pack

**Файл:** `app/bot/main.py`, `buy_pack`

Порядок: первый блок создаёт/получает пользователя и пакет и выходит (commit); второй блок только пишет телеметрию; затем `send_invoice`. Между первым блоком и отправкой счёта пользователь или пакет теоретически могут быть изменены/отключены в другой сессии. Для редкого legacy-пути риск низкий, но при желании можно перейти на одну сессию (получить user/pack, track, send_invoice, затем commit), чтобы не опираться на «уже закоммиченное» состояние между блоками.

## 3. Logic problems

### 3.1 Карточка «Успешных» на дашборде — 14 дней, остальные карточки — 24 часа

**Файл:** `admin-frontend/src/pages/DashboardPage.tsx`

Для «Успешных» подставляется `jobsByStatus.SUCCEEDED` из `data14d` (14 дней), для «За 24 ч», «В очереди» и т.д. используется `data` (24 ч). Получается смешение окон в одном ряду карточек.

**Рекомендация:** либо везде в карточках использовать одно окно (например 24 ч), либо явно подписать под карточкой «Успешных» «за 14 д», чтобы не вводить в заблуждение.

---

### 3.2 pay_initiated до send_invoice: счёт мог не отправиться

Во всех точках мы делаем `track("pay_initiated")`, затем `send_invoice` / `create_invoice_link`. Если после track() произойдёт исключение (сеть, Telegram API), событие «нажата оплата» уже будет в БД, а счёт пользователь не увидит. В воронке это даст «лишние» pay_initiated относительно pay_success.

**Оценка:** для метрики «попытка нажать оплату» это допустимо; для «точно открыли окно оплаты» — нет. Текущая трактовка «нажата оплата» ближе к первому. Можно оставить как есть, но зафиксировать в описании воронки.

## 4. Edge cases missed

### 4.1 unlock_photo: callback.data с лишними двоеточиями

**Файл:** `app/bot/main.py`, `unlock_photo`

Используется `job_id = callback.data.split(":", 1)[1]`. Если payload будет вида `unlock:job:123`, в job_id попадёт `job:123`. Нужно убедиться, что в callback_data нигде не формируется больше одного двоеточия после префикса, иначе job_id станет невалидным.

**Рекомендация:** проверить все места, где создаётся `unlock:{job_id}`; при необходимости парсить только последнюю часть или один раз split по первому `:`.

---

### 4.2 Ошибка при track() в середине блока из нескольких вызовов

**Файл:** `app/bot/main.py`, например pay_method_yoomoney (три вызова track подряд).

Если второй вызов (pay_initiated) упадёт по БД, третий всё равно выполнится в той же сессии после rollback. Состояние БД после блока: часть событий есть, части нет. Для аналитики это приемлемо; для строгой консистентности можно объединить три записи в одну транзакцию (один commit после всех add) — но тогда при любой ошибке откатятся все три. Текущее поведение можно оставить, если явно считать события независимыми.

---

### 4.3 Отсутствие лимита на window_days в product-metrics и product-funnel

**Файл:** `app/api/routes/admin.py`

`telemetry_product_metrics(window_days=Query(7, ge=1))` и часть других эндпоинтов не ограничивают верхнее значение. Большой window_days увеличивает нагрузку на БД. Имеет смысл ввести `le=90` или аналогичный лимит везде, где есть window_days/window_hours.
