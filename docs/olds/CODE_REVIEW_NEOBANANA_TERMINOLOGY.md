## 2. High-risk issues

**2.1 Миграция 058 не создаёт записи, только обновляет**

- `UPDATE telegram_message_templates SET value = '...' WHERE key = '...'` не вставляет строки. Если таблица пустая (новый инстанс без seed шаблонов), миграция ничего не добавит. Поведение корректное: в коде используются дефолты из `defaults.py`, но при наличии в БД старых ключей с другими именами или при ручном добавлении ключей после миграции они не обновятся.
- **Риск:** На уже работающих инстансах с кастомными шаблонами миграция перезапишет value по ключу — админские правки потеряются. Это заложено в комментарий миграции; для прода лучше иметь чек-лист или миграцию с условием `WHERE value = old_value`.

**Конкретика по БД (что именно перезаписывается):**

Миграция 058 выполняет **UPDATE** только по строкам, у которых уже есть запись с указанным `key`. Вставлять новые строки (INSERT) миграция **не** делает.

| Ключ в `telegram_message_templates` | Что делает миграция |
|-------------------------------------|----------------------|
| `start.welcome_text` | Перезаписывает `value` на текст приветствия NeoBanana |
| `menu.btn.shop` | Перезаписывает на «🛒 Купить пакет» |
| `shop.header` | Перезаписывает заголовок магазина |
| `shop.unavailable` | Перезаписывает на «Пакеты временно недоступны.» |
| `errors.job_not_found` | Перезаписывает на «Фото не найдено.» |
| `success.unlock_caption` | Перезаписывает подпись после разблокировки |
| `unlock.invoice_description` | Перезаписывает описание инвойса unlock |
| `payment.pack_success` | Перезаписывает сообщение после успешной оплаты пакета |
| `cmd.terms` | Перезаписывает текст /terms |
| `errors.choose_new_photo` | Перезаписывает на «Для этого сценария выберите новое фото.» |

- **Если в БД есть строка с таким `key`** — её поле `value` (и `updated_at`) будет заменено на новый текст. Любая кастомизация админа в «Сообщения бота» по этим ключам **потеряется**.
- **Если в БД нет строки с таким `key`** — UPDATE затронет 0 строк, ошибки не будет. Бот продолжит брать текст из `defaults.py` для этого ключа.
- **Итог:** перезаписываются ровно эти 10 ключей, и только там, где они уже существуют в таблице. Остальные ключи (help, profile, bank_transfer и т.д.) миграция не трогает.

**2.2 Кнопка магазина в Reply-клавиатуре и фильтр обработчика**

- Обработчик: `lambda m: (m.text or "").strip() == t("menu.btn.shop", "🛒 Купить пакет")`. Вызов `t()` выполняется при оценке фильтра (при приходе сообщения), а не при регистрации роутера — сравнение идёт с актуальным значением шаблона. Регрессии нет.
- Если в БД будет пустой или другой ключ `menu.btn.shop`, дефолт «🛒 Купить пакет» сработает; после миграции значение в БД совпадает с дефолтом. Риск только при ручном изменении текста кнопки в админке без учёта того, что тот же текст должен совпадать с фильтром.

**2.3 Жёстко заданный текст «📋 В меню» в воркерах**

- **deliver_hd.py**, **balance_tariffs.py:** Кнопка «📋 В меню» захардкожена, без `t("nav.btn.menu", ...)`. При смене формулировки в шаблонах/БД в боте и в этих местах текст разъедется. Риск средний (консистентность формулировок).

## 3. Logic problems

**3.1 Сценарий «коллекция» в session_status по-прежнему показывает счётчик 4K**

- **main.py, ~5649–5654:** Для `is_collection` выводится «Выбери до {hd_limit} 4K — осталось: {hd_rem}» и «отмечено для 4K». По ТЗ в пользовательском UI один счётчик — «Осталось фото: {N}», отдельный «Доступно 4K» убран. Для обычной сессии это сделано; для коллекции 4K-счётчик оставлен осознанно под сценарий выбора 4K. С точки зрения единой терминологии это смешение: в одном экране «Осталось фото», в другом — «4K осталось». Решение: либо привести к одному счётчику и формулировкам, либо явно зафиксировать в ТЗ исключение для экрана коллекции.

**3.2 profile.body в defaults не используется в коде профиля**

- В main.py текст профиля собирается вручную (Пакет, Осталось фото), шаблон `profile.body` с плейсхолдерами `{token_balance}` и т.д. не вызывается. Не баг, но при будущем переходе на шаблон нужно учесть, что сейчас в defaults «Осталось фото» уже приведено к терминологии; переменные должны совпадать с тем, что передаётся при рендере.

## 4. Edge cases missed

**4.1 Пустой telegram_id в _show_shop**

- `telegram_id = str(message.from_user.id) if message.from_user else ""`. При `message.from_user is None` получится `telegram_id == ""`. В `build_balance_tariffs_message` передаётся в `get_balance_line(db, "")` и `user_svc.get_by_telegram_id("")` — вернётся None, session = None, has_session = False, body без баланса. Падения нет, но для пустого telegram_id магазин откроется без строки баланса. В нормальных условиях от Telegram всегда приходит from_user.

**4.2 get_balance_line вызывается только при has_session**

- При отсутствии сессии строка «Купите пакет, чтобы начать» не показывается в шапке магазина — показывается только body («Выберите пакет 👇» и список). Это согласовано с текущей логикой (баланс показываем только при активной сессии).

**4.3 Кеш шаблонов (runtime_templates)**

- TTL 60 с и инвалидация по Redis. После миграции 058 без рестарта бота кеш может ещё держать старые value до истечения TTL или до инвалидации. Для деплоя типично: миграция → рестарт, так что на практике ок.
