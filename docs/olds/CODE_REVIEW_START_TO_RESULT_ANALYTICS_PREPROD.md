# Code review: аналитика «время и шаги от старта до результата» (перед продом)

## Обзор фичи

- **Бэкенд:** в `GET /admin/telemetry/product-metrics-v2` добавлен расчёт `avg_time_start_to_result_sec` и `avg_steps_start_to_result` по событиям `bot_started` и `favorite_selected` из `product_events` (одно значение на пользователя: первый старт, первый результат в окне).
- **Фронт:** в блоке «Пользовательские метрики» (Overview) добавлены две карточки: «Среднее время до результата» и «Среднее шагов до результата».

---

## 1. Critical bugs

**Не найдено.** Логика выбора пользователей (есть оба события, result_ts >= start_ts), расчёт длительности и шагов, а также отображение на фронте согласованы с планом. Нет явных падений или неверных типов.

---

## 2. High-risk issues

### 2.1 Нагрузка при большом числе пользователей и событий

**Файл:** `app/api/routes/admin.py`, блок расчёта avg_time / avg_steps.

- `start_result_rows` загружает **все** события `bot_started` и `favorite_selected` за окно (до 90 дней) без лимита.
- `all_events_in_window` загружает **все** события по `user_id.in_(user_ids)` и `timestamp >= since` — при тысячах пользователей и десятках событий на пользователя объём строк может быть очень большим.
- Риск: при окне 30–90 дней и высокой активности — десятки тысяч строк в памяти, возможны таймауты или рост потребления памяти API.

**Рекомендация:** ввести лимит на число пользователей (например, брать не более N пользователей с обоими событиями, по первым по времени), либо считать шаги одним агрегирующим запросом в БД (подзапрос/группировка), чтобы не материализовать все события в Python.

### 2.2 Нет лимита на `window_days` в product-metrics-v2

В эндпоинте используется `window_days: int = Query(7, ge=1, le=90)`. Лимит уже есть (`le=90`), риск из п. 2.1 связан не с параметром, а с объёмом данных при допустимом окне.

---

## 3. Logic problems

### 3.1 Семантика «первый старт / первый результат в окне»

Метрика считает по каждому пользователю **первый** `bot_started` и **первый** `favorite_selected` в выбранном окне. Если пользователь за 7 дней несколько раз заходил и несколько раз выбирал избранное, в расчёт попадает один «путь»: от самого раннего старта до самого раннего выбора в этом окне. Это может объединять разные сессии (старт в понедельник, выбор в пятницу). Для «среднего времени от старта до результата» такая трактовка допустима, но стоит иметь в виду: метрика не «среднее по сессиям», а «среднее по пользователям (один отрезок на пользователя в окне)».

Регрессий и противоречий с планом нет.

---

## 4. Edge cases missed

### 4.1 Пустое множество пользователей

При отсутствии пользователей с обоими событиями в окне `valid_users` пустой, возвращаются `avg_time_start_to_result_sec = None` и `avg_steps_start_to_result = None`. На фронте проверка `!= null` даёт отображение «—». Обработано корректно.

### 4.2 Часовые пояса

`ProductEvent.timestamp` — `DateTime(timezone=True)`, сравнения и `(result_ts - start_ts).total_seconds()` выполняются с учётом времени. Дополнительных edge cases по таймзонам не выявлено.

### 4.3 Целочисленность и формат шагов

При `avg_steps_start_to_result === 0` на фронте выводится «0» (целое). При дробном значении — один знак после запятой. Отрицательные значения на бэкенде не формируются (фильтр result_ts >= start_ts). Корректно.

### 4.4 Очень большое среднее время

При времени > 60 сек отображаются минуты (`toFixed(1)`). При времени в несколько дней значение будет большим (например, «1440.0 мин») — визуально неудобно, но не баг. При желании можно добавить отображение в часах/днях для больших значений.

## 5. What must be fixed before release

| Статус     | Действие |
|------------|----------|
| Сделано    | Ограничение до 2000 пользователей (с наибольшим result_ts — наиболее свежие результаты в окне); при большем числе пользователей усреднение считается по этой выборке. |
| Сделано    | На фронте время отображается: сек (< 1 мин), мин (< 1 ч), ч (< 24 ч), дн. (≥ 24 ч). |

Критичных исправлений перед выкатом не требовалось.

---

## 6. Final verdict

**Ready** для прода при текущей нагрузке.

Логика расчёта и отображения соответствует плану, граничные случаи (нет данных, нули, время в сек/мин) обработаны. При значительном росте числа пользователей и событий стоит выполнить рекомендации из п. 2.1 и 5, чтобы эндпоинт product-metrics-v2 оставался стабильным по времени ответа и памяти.
